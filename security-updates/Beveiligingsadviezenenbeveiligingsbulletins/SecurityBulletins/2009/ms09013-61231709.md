Security Bulletin

Microsoft-beveiligingsbulletin MS09-013 - Kritiek
=================================================

Door beveiligingslekken in Windows HTTP Services kan externe code worden uitgevoerd (960803)
--------------------------------------------------------------------------------------------

Gepubliceerd: dinsdag 14 april 2009 | Bijgewerkt: woensdag 29 april 2009

**Versie:** 1.1

### Algemene informatie

#### Samenvatting

Met deze beveiligingsupdate wordt een openbaar gemaakt beveiligingslek en twee privé gemelde beveiligingslekken in Microsoft Windows HTTP Services (WinHTTP) opgelost. Door het ernstigste beveiligingslek kan externe code worden uitgevoerd. Een aanvaller die misbruik weet te maken van dit beveiligingslek, kan volledige controle krijgen over een systeem waarin dit probleem optreedt. De aanvaller kan vervolgens programma's installeren, gegevens bekijken, wijzigen of wissen, of nieuwe accounts met volledige gebruikersrechten maken. Gebruikers met accounts waarvoor minder gebruikersrechten op het systeem zijn ingesteld, lopen minder risico dan gebruikers die met beheerdersrechten werken.

Dit is een kritieke beveiligingsupdate voor alle ondersteunde edities van Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista en Windows Server 2008. Zie de subsectie **Software waarin dit probleem optreedt en software waarin dit probleem niet optreedt** in deze sectie voor meer informatie.

De beveiligingsupdate lost de beveiligingslekken op door de manier te wijzigen waarop Windows HTTP Services fouten verwerkt en certificaten valideert en door te zorgen dat Windows HTTP Services beveiligingsmechanismen voor hergebruik van NTLM-referenties op de juiste wijze gebruikt. Zie de subsectie Veelgestelde vragen over het specifieke beveiligingslek in de volgende sectie, **Informatie over het beveiligingslek**, voor meer informatie over de beveiligingslekken.

**Aanbeveling.** De meeste klanten hebben automatische updates ingeschakeld en hoeven geen actie te ondernemen omdat deze beveiligingsupdate automatisch wordt gedownload en geïnstalleerd. Klanten die automatische updates niet hebben ingeschakeld, moeten controleren op updates en deze update handmatig installeren. Zie [Microsoft Knowledge Base-artikel 294871](http://support.microsoft.com/kb/294871) voor informatie over specifieke configuratieopties voor het uitvoeren van automatische updates.

Als het gaat om beheerders en bedrijfinstallaties of eindgebruikers die deze beveiligingsupdate handmatig willen installeren, adviseert Microsoft klanten de update onmiddellijk toe te passen met behulp van de software voor updatebeheer of door op updates te controleren met de service [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Zie ook de sectie **Hulpmiddelen en richtlijnen voor detecteren en implementeren**, verderop in dit bulletin.

**Bekende problemen.** In [Microsoft Knowledge Base-artikel 960803](http://support.microsoft.com/kb/960803) worden bekende problemen beschreven die klanten mogelijk ondervinden bij de installatie van deze beveiligingsupdate. In het artikel worden ook de aanbevolen oplossingen voor deze problemen toegelicht.

Ja, om het probleem op te lossen dat wordt beschreven in CVE-2009-0089, heeft Microsoft een nieuwe registersleutel geïmplementeerd (UseFQDNFallbackForCnCompare) onder Software\\Microsoft\\Windows\\CurrentVersion\\Internet Settings\\WinHTTP, waarmee het gedrag van WinHTTP wordt beheerd bij het valideren van de FQDN-naam (Fully Qualified Domain Name) van een host in de DN-vermelding (Distinguished Name) van een certificaat.

Standaard bestaat deze registersleutel niet, waardoor WinHTTP volledig controleert of de DN-naam (Distinguished Name) in het certificaat identiek is aan de DN-naam waarmee de verbinding is gestart. Dit is het minder veilige gedrag, dat standaard was ingebouwd voordat deze beveiligingsupdate werd geïnstalleerd.

Beveiligingslek in Windows HTTP Services met betrekking tot negatieve overloop van gehele getallen - CVE-2009-0086  
------------------------------------------------------------------------------------------------------------------
  
<span></span>
Er is een beveiligingslek met betrekking tot het uitvoeren van externe code aanwezig in de manier waarop Windows HTTP Services specifieke waarden verwerkt die door een externe webserver worden geretourneerd. Een aanvaller die misbruik weet te maken van dit beveiligingslek, kan volledige controle krijgen over een systeem waarin dit probleem optreedt. Een aanvaller zou dan programma's kunnen installeren; gegevens kunnen weergeven, wijzigen en verwijderen of nieuwe accounts met dezelfde gebruikersrechten als de service of toepassing die de WinHTTP-API aanroept kunnen maken om een verbinding met de webserver van de aanvaller tot stand te brengen.
  
Zie [CVE-2009-0086](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0086) als u dit beveiligingslek wilt weergeven als standaardvermelding in de lijst met veelvoorkomende beveiligingslekken en blootstellingen.
  
#### Beperkende factoren voor het beveiligingslek in Windows HTTP Services met betrekking tot negatieve overloop van gehele getallen - CVE-2009-0086
  
Met beperkende factoren wordt verwezen naar een instelling, de algehele configuratie of algemene tips voor een standaardstatus waarmee de ernst van misbruik via een beveiligingslek kan worden verminderd. De volgende beperkende factoren kunnen in uw situatie van pas komen:
  
-   Geen enkele ondersteunde editie van Internet Explorer wordt door dit beveiligingslek getroffen omdat Windows HTTP Services niet door de software wordt gebruikt.
  
#### Tijdelijke oplossingen voor het beveiligingslek in Windows HTTP Services met betrekking tot negatieve overloop van gehele getallen - CVE-2009-0086
  
Microsoft heeft nog geen oplossingen voor dit beveiligingslek.
  
#### Veelgestelde vragen voor het beveiligingslek in Windows HTTP Services met betrekking tot negatieve overloop van gehele getallen - CVE-2009-0086
  
**Wat is de omvang van het beveiligingslek?**    
Dit beveiligingslek heeft betrekking op de uitvoering van externe code. Een aanvaller die misbruik weet te maken van dit beveiligingslek, kan volledige controle krijgen over een systeem waarin dit probleem optreedt. Een aanvaller zou dan programma's kunnen installeren; gegevens kunnen weergeven, wijzigen en verwijderen of nieuwe accounts met dezelfde gebruikersrechten als de service of toepassing die de WinHTTP-API aanroept kunnen maken om een verbinding met de webserver van de aanvaller tot stand te brengen.
  
Dit beveiligingslek is van toepassing op Windows-onderdelen en toepassingen van derden die gebruikmaken van Windows HTTP Services en zijn geconfigureerd om verbinding te maken met externe, niet-vertrouwde webservers.
  
**Waardoor wordt het beveiligingslek veroorzaakt?**    
Het beveiligingslek wordt veroorzaakt doordat een parameter die door een externe webserver wordt geretourneerd, niet wordt gevalideerd. Dit kan leiden tot uitvoering van willekeurige code.
  
**Wat is Windows HTTP Services?**    
Windows HTTP Services (WinHTTP) voorziet ontwikkelaars van een HTTP-client-API (Application Programming Interface) voor het verzenden van aanvragen naar webservers via het HTTP-protocol. WinHTTP kan door zowel Microsoft Windows-onderdelen als software van derden worden gebruikt. WinHTTP wordt onder andere gebruikt door de UPnP-service (Universal Plug and Play).
  
**Wat is Universal Plug and Play?**    
Universal Plug and Play (UPnP) is een voorziening waarmee apparaten in een netwerk andere apparaten kunnen ontdekken en kunnen bepalen hoe ze ermee moeten communiceren. UPnP is het best te vergelijken met Plug and Play (PnP), waar de meeste Windows-gebruikers reeds vertrouwd mee zijn. Met PnP kan het besturingssysteem nieuwe hardware vinden wanneer u deze in een systeem installeert. Wanneer u bijvoorbeeld een nieuwe muis op uw computer installeert, kan Windows deze dankzij PnP vinden. Vervolgens kan Windows de benodigde stuurprogramma's laden en de nieuwe muis in gebruik nemen. UPnP biedt dezelfde mogelijkheden ook voor apparaten in een netwerk, in plaats van alleen het lokale systeem zelf. Dankzij UPnP kunnen computers andere apparaten in het netwerk vinden en bepalen hoe ze ermee moeten communiceren. En computer kan UPnP bijvoorbeeld gebruiken om te bepalen of er beschikbare printers in het netwerk zijn en hoe ermee moet worden gecommuniceerd.
  
**Wat is Simple Service Discovery Protocol (SSDP)?**    
Apparaatontdekking is het proces waardoor computers met UPnP-ondersteuning nieuwe beschikbare apparaten kunnen vinden en kunnen bepalen hoe services van die apparaten moeten worden aangevraagd. Wanneer een computer met UPnP-ondersteuning wordt opgestart, zijn er mogelijk al bruikbare apparaten in het netwerk. Om vast te stellen of dit het geval is, verzendt de computer een uitzendingsaanvraag (M-SEARCH-instructie) waarop elk apparaat in het subnet met UPnP-ondersteuning onmiddellijk kan reageren met communicatie-instructies.
  
Wanneer op vergelijkbare wijze een apparaat met ondersteuning voor UPnP (bijvoorbeeld een UPnP-printer) wordt opgestart, zijn er mogelijk al computers met UPnP-ondersteuning in het netwerk die gebruik willen maken van het apparaat. Het apparaat zendt een bericht (een NOTIFY-instructie) uit naar alle computers binnen het subnet om de aanwezigheid in het netwerk aan te kondigen en computers uit te nodigen gebruik te maken van de services van het apparaat.
  
**Met welk doel kan een aanvaller het beveiligingslek misbruiken?**    
Een aanvaller die dit beveiligingslek weet te misbruiken, kan dezelfde gebruikersrechten als de service of toepassing die de WinHTTP-API aanroept krijgen om een verbinding met de webserver van de aanvaller tot stand te brengen. Toepassingen die worden uitgevoerd met accounts waarvoor minder gebruikersrechten op het systeem zijn ingesteld, lopen minder risico dan toepassingen die worden uitgevoerd met accounts waarvoor beheerdersrechten zijn ingesteld.
  
**Hoe kan een aanvaller dit beveiligingslek misbruiken?**    
Een aanvaller zou een programma dat gebruikmaakt van Windows HTTP Services ertoe moeten dwingen of overhalen verbinding te maken met de schadelijke webserver van de aanvaller. Het is belangrijk om te weten dat de UPnP-service (Universal Plug and Play) gebruikmaakt van de WinHTTP-bibliotheken. Wanneer deze service wordt ingeschakeld, kunnen kwaadwillende gebruikers op het lokale subnet SSDP-aanvragen beantwoorden en de UPnP-service via WinHTTP verbinding laten maken met een schadelijke host die het beveiligingslek vervolgens kan misbruiken. De UPnP-service wordt uitgevoerd onder de account LocalService. Dit is een account met beperkte bevoegdheden in het systeem.
  
**Voor welke systemen vormt dit beveiligingslek het grootste risico?**    
Voor werkstations en terminalservers is het risico het grootst. Servers zouden eventueel meer gevaar kunnen lopen als beheerders toestaan dat gebruikers zich op servers kunnen aanmelden en programma's kunnen uitvoeren. Dit is echter volkomen in strijd met de aanbevolen richtlijnen voor gebruik.

#### Veelgestelde vragen voor het beveiligingslek in Windows HTTP Services met betrekking tot niet-overeenkomende certificaatnamen - CVE-2009-0089
  
**Wat is de omvang van het beveiligingslek?**    
Dit is een beveiligingslek met betrekking tot vervalsing (spoofing) dat in Windows HTTP Services bestaat. Als het beveiligingslek wordt gecombineerd met specifieke andere aanvallen, zoals DNS-vervalsing (spoofing), kan een aanvaller het digitale certificaat van een website vervalsen voor een willekeurige toepassing die gebruikmaakt van Windows HTTP Services.
  
**Waardoor wordt het beveiligingslek veroorzaakt?**    
Wanneer een toepassing Windows HTTP Services aanroept om een verbinding met een externe webserver in te stellen, valideert WinHTTP alleen de FQDN-naam (Fully Qualified Domain Name) van de URL tegen het certificaat van de oorspronkelijke website waarmee de host verbinding probeerde te maken. Specifieke combinaties van DNS-vervalsing (spoofing) kunnen een verbinding doorsturen naar een andere webserver die een geldig certificaat voor die specifieke host heeft, maar niet voor de webpagina waarmee WinHTTP de verbinding heeft begonnen. Als DNS-vervalsing met dit beveiligingslek wordt gecombineerd, kan WinHTTP het certificaat voor de externe webserver onterecht als acceptabel beschouwen. Als WinHTTP van de toepassing afhankelijk is voor gebruikersinteractie, wordt de gebruiker mogelijk niet door de toepassing gewaarschuwd voor deze afwijking.
  
**Wat is Windows HTTP Services?**    
Windows HTTP Services (WinHTTP) voorziet ontwikkelaars van een HTTP-client-API (Application Programming Interface) voor het verzenden van aanvragen naar andere HTTP-servers via het HTTP-protocol.
  
**Wat zijn digitale certificaten?**    
Het digitale certificaat is een gemeenschappelijke referentie die wordt gebruikt om een identiteit te verifiëren. Een vertrouwde organisatie wijst een certificaat toe aan een individu of een entiteit, het onderwerp van dat certificaat. De vertrouwde organisatie die het certificaat uitgeeft, is een certificeringsinstantie (Certification Authority, CA) en fungeert als uitgever van het certificaat. Een betrouwbare certificeringsinstantie zal een certificaat alleen na het verifiëren van de identiteit van het onderwerp van het certificaat uitgeven.
  
**Wat is DNS-vervalsing (spoofing)?**    
Domain Name System (DNS) is een van de industriestandaardprotocollen waaruit TCP/IP bestaat. DNS wordt geïmplementeerd met twee softwarecomponenten: de DNS-server en de DNS-client (of resolver). Beide onderdelen worden als achtergrondservicetoepassingen uitgevoerd. Netwerkbronnen worden geïdentificeerd door numerieke IP-adressen, maar deze IP-adressen zijn moeilijk te onthouden voor netwerkgebruikers. De DNS-database bevat records waarmee gebruikersvriendelijke alfanumerieke namen voor netwerkbronnen worden toegewezen aan de IP-adressen die door deze bronnen worden gebruikt voor communicatie. Op deze manier fungeert DNS als een instructieapparaat, waardoor netwerkgebruikers netwerkbronnen eenvoudiger kunnen onthouden. Lees het TechNet-artikel [What is DNS](http://technet.microsoft.com/en-us/library/cc787921.aspx) (Engelstalig) voor meer informatie en voor logische diagrammen waarin wordt uitgelegd hoe DNS bij andere Windows-technologieën past.
  
DNS-vervalsing (spoofing) is een brede categorie aanvallen waarbij op een punt binnen het DNS-systeem onjuiste DNS-antwoorden worden geïntroduceerd door een derde die normaal niet verantwoordelijk is voor die antwoorden. Zo'n aanval kan ertoe leiden dat een domein of hostnaam wordt omgezet in een ander IP-adres dan de eigenaars van de naam voor ogen hadden. Op deze manier kan een host worden uitgelokt verbinding te maken met een mogelijk schadelijke server van een derde.
  
**Met welk doel kan een aanvaller het beveiligingslek misbruiken?**    
Een aanvaller zou een beveiligde website kunnen imiteren en via Windows HTTP Services schadelijke inhoud kunnen verstrekken aan de toepassing, die de inhoud vervolgens zou vertrouwen alsof deze afkomstig zou zijn van een beveiligde website.
  
**Hoe kan een aanvaller dit beveiligingslek misbruiken?**    
Een aanvaller zou dit beveiligingslek moeten combineren met DNS-vervalsing (spoofing) of een vergelijkbare aanval om de uitgaande verbinding naar een server over te nemen. Hij zou dan een certificaat aanbieden dat geldig is voor zijn website, dat onterecht zou worden gevalideerd door Windows HTTP Services en als geldig zou worden beschouwd voor de doelhostwebsite waarmee de gebruiker oorspronkelijk verbinding probeerde te maken.

Voor werkstations en terminalservers is het risico het grootst. Servers zouden eventueel meer gevaar kunnen lopen als beheerders toestaan dat gebruikers zich op servers kunnen aanmelden en programma's kunnen uitvoeren. Dit is echter volkomen in strijd met de aanbevolen richtlijnen voor gebruik.

**Controleren of de update is toegepast**

-   **Microsoft Baseline Security Analyzer**

    Gebruik het hulpprogramma Microsoft Baseline Security Analyzer (MBSA) om te controleren of de beveiligingsupdate is toegepast op het systeem waarvoor het probleem geldt. Zie de sectie **Hulpmiddelen en richtlijnen voor detecteren en implementeren** eerder in dit bulletin voor meer informatie.

-   **Verificatie van bestandsversie**

    Omdat er verschillende versies zijn van Microsoft Windows, kunnen de volgende stappen voor uw computer enigszins afwijken. Als dit het geval is, raadpleeg dan de productdocumentatie om deze stappen uit te voeren.

    1.  Klik op **Start** en klik vervolgens op **Zoeken**.
    2.  Klik in het venster **Zoekresultaten** op **Alle bestanden en mappen** onder **Zoekassistent**.
    3.  Typ in het vak **De volledige of gedeeltelijke bestandsnaam** een bestandsnaam uit de betreffende bestandsinformatietabel en klik vervolgens op **Zoeken**.

    5.  Bepaal op het tabblad **Versie** de versie van het bestand dat op uw systeem is geïnstalleerd door de versie te vergelijken met de versie in de betreffende bestandsinformatietabel.
        **Opmerking** Andere kenmerken dan de bestandsversie kunnen tijdens de installatie worden veranderd. Vergelijking van andere bestandskenmerken met de gegevens in de bestandsinformatietabel vormt geen ondersteunde methode om te controleren of de update is toegepast. Bovendien kunnen in bepaalde gevallen de namen van bestanden tijdens installatie worden gewijzigd. Als de bestands- of versiegegevens niet aanwezig zijn, gebruikt u een van de andere beschikbare methoden om te controleren of de update is geïnstalleerd.

    Omdat er verschillende versies zijn van Microsoft Windows, kunnen de volgende stappen voor uw computer enigszins afwijken. Als dit het geval is, raadpleeg dan de productdocumentatie om deze stappen uit te voeren.

    1.  Klik op **Start** en klik vervolgens op **Zoeken**.
    2.  Klik in het venster **Zoekresultaten** op **Alle bestanden en mappen** onder **Zoekassistent**.

        **Opmerking** Bepaalde bestanden die in de bestandsinformatietabel staan vermeld, worden mogelijk niet geïnstalleerd. Dit is afhankelijk van de editie van het besturingssysteem en de programma's die op uw systeem zijn geïnstalleerd.

        **Opmerking** Andere kenmerken dan de bestandsversie kunnen tijdens de installatie worden veranderd. Vergelijking van andere bestandskenmerken met de gegevens in de bestandsinformatietabel vormt geen ondersteunde methode om te controleren of de update is toegepast.
