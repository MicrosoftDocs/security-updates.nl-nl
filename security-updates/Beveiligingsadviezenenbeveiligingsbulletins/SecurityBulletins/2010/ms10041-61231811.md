Security Bulletin

Microsoft-beveiligingsbulletin MS10-041 - Belangrijk
====================================================

Door een beveiligingslek in Microsoft .NET Framework kan vervalsing plaatsvinden (981343)
-----------------------------------------------------------------------------------------

Gepubliceerd: donderdag 10 juni 2010 | Bijgewerkt: dinsdag 22 juni 2010

**Versie:** 1.2

### Algemene informatie

#### Samenvatting

Met deze beveiligingsupdate wordt een openbaar gemeld beveiligingslek in Microsoft .NET Framework opgelost. Door het beveiligingslek kunnen ongemerkt gegevens van ondertekende XML-inhoud worden vervalst. In aangepaste toepassingen is de invloed van het beveiligingslek afhankelijk van de manier waarop de ondertekende inhoud wordt gebruikt in de specifieke toepassing. Dit beveiligingslek treedt niet op in scenario's waarin ondertekende XML-berichten worden verzonden via een beveiligd kanaal (zoals SSL).

Het prioriteitsniveau van deze beveiligingsupdate is Belangrijk voor alle getroffen edities van Microsoft .NET Framework voor Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista, Windows Server 2008, Windows 7 en Windows Server 2008 R2. Zie de subsectie **Software waarin dit probleem optreedt en software waarin dit probleem niet optreedt** in deze sectie voor meer informatie.

De beveiligingsupdate lost het beveiligingslek op door de manier te wijzigen waarop de XMLDsig-aanbeveling is geïmplementeerd in Microsoft .NET Framework. Zie de subsectie Veelgestelde vragen over het specifieke beveiligingslek in de volgende sectie**,Informatie over het beveiligingslek**, voor meer informatie over het beveiligingslek.

**Aanbeveling.**  De meeste klanten hebben automatische updates ingeschakeld en hoeven geen actie te ondernemen omdat deze beveiligingsupdate automatisch wordt gedownload en geïnstalleerd. Klanten die automatische updates niet hebben ingeschakeld, moeten controleren op updates en deze update handmatig installeren. Zie [Microsoft Knowledge Base-artikel 294871](http://support.microsoft.com/kb/294871) voor informatie over specifieke configuratieopties voor het uitvoeren van automatische updates.

Als het gaat om beheerders en bedrijfinstallaties of eindgebruikers die deze beveiligingsupdate handmatig willen installeren, adviseert Microsoft klanten de update zo snel mogelijk toe te passen met behulp van de software voor updatebeheer of door op updates te controleren met de service [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Zie ook de sectie **Hulpmiddelen en richtlijnen voor detecteren en implementeren**, verderop in dit bulletin.

**Bekende problemen.** In [Microsoft Knowledge Base-artikel 981343](http://support.microsoft.com/kb/981343) worden bekende problemen beschreven die klanten mogelijk ondervinden bij de installatie van deze beveiligingsupdate. In het artikel worden ook de aanbevolen oplossingen voor deze problemen toegelicht.

#### Software waarin dit probleem optreedt en software waarin dit probleem niet optreedt

De volgende software is getest om te bepalen in welke versies of edities dit probleem optreedt. Andere versies of edities hebben het einde van hun ondersteuningscyclus bereikt of hebben geen last van dit beveiligingslek. Ga naar [Microsoft Support Lifecycle](http://go.microsoft.com/fwlink/?linkid=21742) om de ondersteuningscyclus voor uw product en versie te bepalen.

Veelgestelde vragen met betrekking tot deze beveiligingsupdate
--------------------------------------------------------------

<span></span>
**Waarom zijn er op 22 juni 2010 wijzigingen in dit bulletin aangebracht? **
Microsoft heeft dit bulletin bijgewerkt met de aankondiging van een wijziging in de detectielogica om een probleem op te lossen waarbij de beveiligingsupdate van 8 juni 2010 voor Microsoft .NET Framework 3.5 (KB982865) ten onrechte werd gezien als "geïnstalleerd" door de Microsoft Baseline Security Analyzer (MBSA) op systemen waarop Microsoft .NET Framework 3.5 Service Pack 1 is geïnstalleerd. Desondanks wordt de toepasselijkheid van de beveiligingsupdate voor Microsoft .NET Framework 2.0 Service Pack 2 en Microsoft .NET Framework 3.5 Service Pack 1 (KB979909) correct gedetecteerd door MBSA. Dit is slechts een detectiewijziging. De wijziging heeft geen invloed op de bestanden van de eerste update. Klanten die hun systemen al hebben bijgewerkt, hoeven deze update niet opnieuw te installeren.

Door deze detectiewijziging kan de update van KB982865 worden weergegeven als een nieuwe update voor klanten met Windows Server Update Services (WSUS). Als u geen regels voor automatische goedkeuring hebt ingesteld, moet u de update mogelijk handmatig opnieuw goedkeuren. Als u de update niet opnieuw goedkeurt, kan de update opnieuw worden aangeboden, ook als de update niet van toepassing is. Zie [Microsoft Knowledge Base-artikel 894199](http://support.microsoft.com/kb/894199) voor meer informatie over deze detectiewijziging.

**Wat is een beveiligingslek met betrekking tot gegevensvervalsing? **
In informatiebeveiliging kan een beveiligingslek met betrekking tot gegevensvervalsing leiden tot een schadelijke bewerking van gegevens. Voorbeelden zijn onbevoegde wijzigingen in blijvende gegevens, bijvoorbeeld gegevens in een database, en de wijziging van gegevens terwijl deze worden verzonden tussen twee computers via een openbaar netwerk, zoals internet. In het geval van dit beveiligingslek wordt een effectieve cryptografische handtekening gebruikt om gegevens digitaal te ondertekenen. Als deze gegevens worden gewijzigd na de ondertekening en de digitale handtekening niet kan worden geverifieerd, duidt dit op gegevensvervalsing. Ook als de handtekening wordt geverifieerd, ondanks dat de gegevens zijn gewijzigd, duidt dit ook op gegevensvervalsing.

**Hoe kan ik vaststellen welke versie van Microsoft .NET Framework is geïnstalleerd?**  
U kunt meerdere versies van .NET Framework op een systeem installeren en uitvoeren en u kunt de versies in willekeurige volgorde installeren. Er zijn diverse manieren om vast te stellen welke versies van .NET Framework zijn geïnstalleerd. Zie [Microsoft Knowledge Base-artikel 318785](http://support.microsoft.com/kb/318785) of het MSDN-artikel [Determining Which Version of the .NET Framework Is Installed](http://msdn.microsoft.com/en-us/library/y549e41e.aspx) voor meer informatie.

**Waarom worden Microsoft .NET Framework 3.5 en Microsoft .NET Framework 3.5 Service Pack 1 getroffen op sommige ondersteunde besturingssystemen?**
Microsoft .Net Framework 3.5 bevat de subonderdelen Microsoft .Net Framework 2.0 Service Pack 1 en Microsoft .Net Framework 3.0 Service Pack 1. Microsoft .Net Framework 3.5 Service Pack 1 bevat de subonderdelen Microsoft .Net Framework 2.0 Service Pack 2 en Microsoft .Net Framework 3.0 Service Pack 2. Op sommige besturingssystemen kunnen Microsoft .NET Framework 3.5 en Microsoft .NET Framework 3.5 Service Pack 1 deze kwetsbare subcomponenten installeren.

De software waarin het probleem optreedt en die in dit beveiligingsbulletin wordt vermeld, is getest om te controleren of het probleem bij deze versies optreedt. Andere edities hebben het einde van hun ondersteuningscyclus bereikt of worden niet ondersteund. **Bevat deze update andere functionaliteitswijzigingen met betrekking tot beveiliging?**  
Ja. Naast de wijzigingen die worden genoemd in de sectie **Informatie over het beveiligingslek** van dit bulletin, bevat deze update een wijziging voor de beveiligingsfunctie voor het valideren van ASP.NET-aanvragen om een probleem op te lossen waardoor een aanvaller een eenvoudige beveiligingsmaatregel kan omzeilen die standaard is ingeschakeld op ASP.NET-websites.

Het probleem wordt veroorzaakt doordat de functie voor het valideren van ASP.NET-aanvragen niet juist controleert op een specifieke tekenreeks. De functie voor het valideren van ASP.NET-aanvragen kan een effectieve validatielaag die niet-vertrouwde invoervariabelen beperkt, niet vervangen. Ontwikkelaars die meer willen weten over de beveiligingsfuncties van ASP.NET voor webtoepassingen, worden verwezen naar het MSDN-artikel [Take Advantage of ASP.NET Built-in Features to Fend Off Web Attacks](http://msdn2.microsoft.com/en-us/library/ms972969.aspx).

Als een website geen effectieve validatielaag heeft om niet-vertrouwde gebruikersinvoer te beperken, kan een aanvaller die misbruik weet te maken van dit beveiligingslek willekeurige inhoud, waaronder ASP.NET-inhoud, op de getroffen website plaatsen. Deze ingrijpende wijziging vormt een beperkende factor voor het privé gemelde probleem.

**Wat is validatie van ASP.NET-aanvragen?**  
Deze ingrijpende wijziging in de functie voor het valideren van ASP.NET-aanvragen voert een eenvoudige invoervalidatie uit op websites met ASP.NET. De functie voor het valideren van ASP.NET-aanvragen kan echter een effectieve validatielaag die niet-vertrouwde invoervariabelen beperkt, niet vervangen. Ontwikkelaars die meer willen weten over de beveiligingsfuncties van ASP.NET voor webtoepassingen, worden verwezen naar de MSDN-artikelen [Take Advantage of ASP.NET Built-in Features to Fend Off Web Attacks](http://msdn2.microsoft.com/en-us/library/ms972969.aspx) en [Do Not Rely on ASP.NET Request Validation](http://msdn.microsoft.com/en-us/library/ms998264.aspx).

**Hoe weet ik of mijn ASP.NET-webtoepassing wordt getroffen door deze ingrijpende wijziging?**  
Dit probleem geldt niet voor door ASP.NET ontwikkelde webtoepassingen die alle niet-vertrouwde invoervariabelen tot een reeks verwachte waarden of tekens beperken. Raadpleeg [Microsoft Knowledge Base-artikel 815155](http://support.microsoft.com/kb/815155) voor meer informatie over de beveiliging van ASP.NET-webtoepassingen.

**Wat is ASP.NET?**  
ASP.NET is een verzameling technologieën binnen Microsoft .NET Framework waarmee ontwikkelaars webtoepassingen en XML-webservices kunnen maken.

In tegenstelling tot traditionele webpagina's, die uit een combinatie van statische HTML en scripting zijn opgebouwd, maakt ASP.NET gebruik van gecompileerde, gebeurtenisgestuurde pagina's. Omdat ASP.NET een webtoepassingsomgeving is en een onderliggende webserver vereist is om basisfunctionaliteit van HTTP te verzorgen, wordt ASP.NET bovenop Internet Information Service (IIS) uitgevoerd. Zie voor meer informatie [de officiële Microsoft ASP.NET-website](http://www.asp.net/).

**Waarom wordt dit probleem behandeld als een ingrijpende veiligheidsmaatregel?**
De validatie van ASP.NET-aanvragen wordt behandeld als een ingrijpende wijziging die kan worden gebruikt als een extra voorzorgsmaatregel, naast de eigen invoervalidatie van de ontwikkelaar. Alleen de ontwikkelaar kan bepalen wat goede invoer is voor een specifieke toepassing. Op ingrijpende veiligheidsmaatregelen mag niet worden vertrouwd, ook al kunnen meerdere van dergelijke lagen in grote mate voorkomen dat aanvallers het betreffende systeem in gevaar kunnen brengen. Als een validatie van ASP.NET-aanvragen een specifieke tekenreeks mist, kan de maximale omvang van het beveiligingslek niet groter zijn dan wanneer deze functie volledig uitgeschakeld zou zijn.

**Wat is ingrijpend?**
In informatiebeveiliging verwijst "ingrijpend" naar een aanpak waarbij meerdere beveiligingslagen worden gebruikt om te zorgen dat aanvallers de beveiliging van een netwerk of systeem niet kunnen ondermijnen.

Beveiligingslek met betrekking tot omzeiling van verificatie van XML-handtekeningen door HMAC-afkapping - CVE-2009-0217
-----------------------------------------------------------------------------------------------------------------------

<span></span>
Er bestaat een beveiligingslek met betrekking tot gegevensvervalsing in Microsoft .NET Framework waardoor een aanvaller ongemerkt ondertekende XML-inhoud kan vervalsen. In aangepaste toepassingen is de omvang van het beveiligingslek afhankelijk van het specifieke gebruik van de toepassing. Dit beveiligingslek treedt niet op in scenario's waarin ondertekende XML-berichten worden verzonden via een beveiligd kanaal (zoals SSL).

Zie [CVE 2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217) als u dit beveiligingslek wilt weergeven als standaardvermelding in de lijst met veelvoorkomende beveiligingslekken en blootstellingen.

#### Beperkende factoren voor het beveiligingslek met betrekking tot omzeiling van verificatie van XML-handtekeningen door HMAC-afkapping - CVE-2009-0217

Met beperkende factoren wordt verwezen naar een instelling, de algehele configuratie of algemene tips voor een standaardstatus waarmee de ernst van misbruik via een beveiligingslek kan worden verminderd. De volgende beperkende factoren kunnen in uw situatie van pas komen:

-   Microsoft Office controleert XML-handtekeningen op een manier die niet door een aanvaller kan worden misbruikt om misbruik te maken van dit beveiligingslek met betrekking tot gegevensvervalsing.

#### Tijdelijke oplossingen voor het beveiligingslek met betrekking tot omzeiling van verificatie van XML-handtekeningen door HMAC-afkapping - CVE-2009-0217

Tijdelijke oplossingen zijn wijzigingen in een instelling of configuratie waardoor het beveiligingslek zelf niet wordt gedicht, maar waarmee aanvalsvectoren wel worden geblokkeerd voordat u de update uitvoert. Microsoft heeft de volgende methoden en statussen om het probleem te omzeilen getest om uit te zoeken of een tijdelijke oplossing functionaliteit vermindert:

-   **Gebruik een veilig kanaal om vertrouwelijke gegevens te verzenden**

    Als uw toepassing vertrouwelijke XML-gegevens of XML-gegevens die kritiek zijn voor de beveiliging van uw toepassing verzendt, gebruik dan een veilig kanaal om deze gegevens te verzenden. Als u bijvoorbeeld Internet Information Server gebruikt, maak dan gebruik van SSL voor vertrouwelijke of kritieke toepassingen.

#### Veelgestelde vragen over het beveiligingslek met betrekking tot omzeiling van verificatie van XML-handtekeningen door HMAC-afkapping - CVE-2009-0217

**Wat is de omvang van het beveiligingslek?**  
Dit is een beveiligingslek met betrekking tot gegevensvervalsing dat kan worden gebruikt om een cryptografische handtekening te omzeilen.

**Waardoor wordt het beveiligingslek veroorzaakt?**  
Het beveiligingslek wordt veroorzaakt door de manier waarop de XMLDsig-aanbeveling (XML Signature Syntax and Processing) van W3C is verwerkt in Microsoft .NET Framework. Zie de W3C-aanbeveling [XML Signature Syntax and Processing (Second Edition)](http://www.w3.org/tr/xmldsig-core/) of de sectie "E03 HMAC truncation (CVE-2009-0217)" in [Errata for XML Signature Second Edition](http://www.w3.org/2008/06/xmldsigcore-errata.html) voor meer informatie over XMLDsig.

**Wat is Microsoft .NET Framework?**  
Microsoft .NET Framework is een onderdeel van het Microsoft Windows-besturingssysteem waarmee softwaretoepassingen en webservices kunnen worden gebouwd en uitgevoerd. Het bevat technologieën voor webservices en webtoepassingen (ASP.NET), toegang tot gegevens (ADO.NET), smart clienttoepassingen (Windows Forms) enzovoort.

**Wat is HMAC?**  
Een Hash-based Message Authentication Code (HMAC) kan worden gebruikt om vast te stellen of een bericht dat is verzonden via een onveilig kanaal is vervalst, op voorwaarde dat de afzender en ontvanger een geheime sleutel delen, door de integriteit en authenticiteit van het bericht te controleren.

**Wat is XML?**  
Extensible Markup Language, of XML, is een indeling voor het beschrijven van gestructureerde gegevens. Extensible Markup Language (XML) is een specificatie van het World Wide Web Consortium (W3C) en een onderdeel van Standard Generalized Markup Language (SGML). Zie de [XML-startpagina](http://msdn.microsoft.com/xml/) van het MSDN Data Developer Center of de W3C-aanbeveling [Extensible Markup Language (XML) 1.0 (Fifth Edition)](http://www.w3.org/tr/rec-xml/) voor meer informatie.

**Met welk doel kan een aanvaller het beveiligingslek misbruiken?**  
Een aanvaller die misbruik weet te maken van dit beveiligingslek kan bepaalde cryptografische handtekeningen omzeilen en daardoor ondertekende XML-inhoud vervalsen zonder dat de ontvanger de wijzigingen opmerkt. Als de oorspronkelijke bedoeling van het bericht volledig wordt gewijzigd, kan dit leiden tot spoofing. In gevallen waarin een ontwikkelaar een toepassing heeft geschreven op basis van met HMAC ondertekende XML-inhoud, hangt de invloed van het beveiligingslek af van het specifieke gebruik van die toepassing.

**Hoe kan een aanvaller dit beveiligingslek misbruiken?**  
Een aanvaller moet speciaal vervaardigde XML-inhoud verzenden naar een kwetsbaar systeem. Microsoft Office controleert XML-handtekeningen op een manier die niet door een aanvaller kan worden misbruikt.

**Voor welke systemen vormt dit beveiligingslek het grootste risico?**  
Het beveiligingslek vormt het grootste risico voor systemen of toepassingen die gebruikmaken van Hash-based Message Authentication Code (HMAC) voor cryptografische bewerkingen. Microsoft Office controleert XML-handtekeningen op een manier die niet door een aanvaller kan worden misbruikt om misbruik te maken van dit beveiligingslek.

**Wat doet de update?**  
De update wijzigt de manier waarop de XMLDsig-aanbeveling is geïmplementeerd in Microsoft .NET Framework.

**Was dit beveiligingslek openbaar gemaakt op het moment dat dit beveiligingsbulletin werd uitgegeven?**  
Ja. Dit beveiligingslek is openbaar gemaakt. Aan dit beveiligingslek is nummer [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217) toegewezen.

**Was aan Microsoft gerapporteerd dat er misbruik was gemaakt van dit beveiligingslek op het moment dat dit beveiligingsbulletin werd uitgegeven?**  
Nee. Microsoft heeft geen informatie ontvangen die erop wees dat dit beveiligingslek was gebruikt om aanvallen uit te voeren op klanten op het moment dat dit beveiligingsbulletin oorspronkelijk werd uitgegeven.

**Controleren of de update is toegepast**

-   **Microsoft Baseline Security Analyzer**

    Gebruik het hulpprogramma Microsoft Baseline Security Analyzer (MBSA) om te controleren of de beveiligingsupdate is toegepast op het systeem waarvoor het probleem geldt. Zie de sectie **Hulpmiddelen en richtlijnen voor detecteren en implementeren** eerder in dit bulletin voor meer informatie.

-   **Verificatie van bestandsversie**

    Omdat er verschillende versies zijn van Microsoft Windows, kunnen de volgende stappen voor uw computer enigszins afwijken. Als dit het geval is, raadpleeg dan de productdocumentatie om deze stappen uit te voeren.

    1.  Klik op **Start** en klik vervolgens op **Zoeken**.
    2.  Klik in het venster **Zoekresultaten** op **Alle bestanden en mappen** onder **Zoekassistent**.
    3.  Typ in het vak **De volledige of gedeeltelijke bestandsnaam** een bestandsnaam uit de betreffende bestandsinformatietabel en klik vervolgens op **Zoeken**.
    4.  Klik in de lijst met bestanden met de rechtermuisknop op een bestandsnaam uit de betreffende bestandsinformatietabel en klik vervolgens op **Eigenschappen**.
        **Opmerking** Bepaalde bestanden die in de bestandsinformatietabel staan vermeld, worden mogelijk niet geïnstalleerd. Dit is afhankelijk van de editie van het besturingssysteem en de programma's die op uw systeem zijn geïnstalleerd.
    5.  Bepaal op het tabblad **Versie** de versie van het bestand dat op uw systeem is geïnstalleerd door de versie te vergelijken met de versie in de betreffende bestandsinformatietabel.
        **Opmerking** Andere kenmerken dan de bestandsversie kunnen tijdens de installatie worden veranderd. Vergelijking van andere bestandskenmerken met de gegevens in de bestandsinformatietabel vormt geen ondersteunde methode om te controleren of de update is toegepast. Bovendien kunnen in bepaalde gevallen de namen van bestanden tijdens installatie worden gewijzigd. Als de bestands- of versiegegevens niet aanwezig zijn, gebruikt u een van de andere beschikbare methoden om te controleren of de update is geïnstalleerd.

-   **Verificatie van registersleutel**

    Aan de hand van de registersleutels in de **Referentietabel** in deze sectie kunt u ook controleren welke bestanden met deze beveiligingsupdate zijn geïnstalleerd.

    Deze registersleutels bevatten mogelijk geen volledige lijst met geïnstalleerde bestanden. Bovendien zijn deze registersleutels mogelijk niet correct als een beheerder of een OEM de beveiligingsupdate integreert in de bronbestanden van de Windows-installatie.

**Controleren of de update is toegepast**

-   **Microsoft Baseline Security Analyzer**

    Gebruik het hulpprogramma Microsoft Baseline Security Analyzer (MBSA) om te controleren of de beveiligingsupdate is toegepast op het systeem waarvoor het probleem geldt. Zie de sectie **Hulpmiddelen en richtlijnen voor detecteren en implementeren** eerder in dit bulletin voor meer informatie.

-   **Verificatie van bestandsversie**

    Omdat er verschillende versies zijn van Microsoft Windows, kunnen de volgende stappen voor uw computer enigszins afwijken. Als dit het geval is, raadpleeg dan de productdocumentatie om deze stappen uit te voeren.

    1.  Klik op **Start** en klik vervolgens op **Zoeken**.
    2.  Klik in het venster **Zoekresultaten** op **Alle bestanden en mappen** onder **Zoekassistent**.
    3.  Typ in het vak **De volledige of gedeeltelijke bestandsnaam** een bestandsnaam uit de betreffende bestandsinformatietabel en klik vervolgens op **Zoeken**.
    4.  Klik in de lijst met bestanden met de rechtermuisknop op een bestandsnaam uit de betreffende bestandsinformatietabel en klik vervolgens op **Eigenschappen**.
        **Opmerking** Bepaalde bestanden die in de bestandsinformatietabel staan vermeld, worden mogelijk niet geïnstalleerd. Dit is afhankelijk van de editie van het besturingssysteem en de programma's die op uw systeem zijn geïnstalleerd.
    5.  Bepaal op het tabblad **Versie** de versie van het bestand dat op uw systeem is geïnstalleerd door de versie te vergelijken met de versie in de betreffende bestandsinformatietabel.
        **Opmerking** Andere kenmerken dan de bestandsversie kunnen tijdens de installatie worden veranderd. Vergelijking van andere bestandskenmerken met de gegevens in de bestandsinformatietabel vormt geen ondersteunde methode om te controleren of de update is toegepast. Bovendien kunnen in bepaalde gevallen de namen van bestanden tijdens installatie worden gewijzigd. Als de bestands- of versiegegevens niet aanwezig zijn, gebruikt u een van de andere beschikbare methoden om te controleren of de update is geïnstalleerd.

-   **Verificatie van registersleutel**

    Aan de hand van de registersleutels in de **Referentietabel** in deze sectie kunt u ook controleren welke bestanden met deze beveiligingsupdate zijn geïnstalleerd.

    Deze registersleutels bevatten mogelijk geen volledige lijst met geïnstalleerde bestanden. Bovendien zijn deze registersleutels mogelijk niet correct als een beheerder of een OEM de beveiligingsupdate integreert in de bronbestanden van de Windows-installatie.

Anders kopieert het installatieprogramma de GDR-versie van de bestanden naar uw systeem. **Controleren of de update is toegepast**

-   **Microsoft Baseline Security Analyzer**

    Gebruik het hulpprogramma Microsoft Baseline Security Analyzer (MBSA) om te controleren of de beveiligingsupdate is toegepast op het systeem waarvoor het probleem geldt. Zie de sectie Hulpmiddelen en richtlijnen voor detecteren en implementeren eerder in dit bulletin voor meer informatie.

-   **Verificatie van bestandsversie**

    Omdat er verschillende versies zijn van Microsoft Windows, kunnen de volgende stappen voor uw computer enigszins afwijken. Als dit het geval is, raadpleeg dan de productdocumentatie om deze stappen uit te voeren.

    1.  Klik op **Start** en klik vervolgens op **Zoeken**.
    2.  Klik in het venster **Zoekresultaten** op **Alle bestanden en mappen** onder **Zoekassistent**.
    3.  Typ in het vak **De volledige of gedeeltelijke bestandsnaam** een bestandsnaam uit de betreffende bestandsinformatietabel en klik vervolgens op **Zoeken**.
    4.  Klik in de lijst met bestanden met de rechtermuisknop op een bestandsnaam uit de betreffende bestandsinformatietabel en klik vervolgens op **Eigenschappen**.
        **Opmerking** Bepaalde bestanden die in de bestandsinformatietabel staan vermeld, worden mogelijk niet geïnstalleerd. Dit is afhankelijk van de editie van het besturingssysteem en de programma's die op uw systeem zijn geïnstalleerd.
    5.  Bepaal op het tabblad **Versie** de versie van het bestand dat op uw systeem is geïnstalleerd door de versie te vergelijken met de versie in de betreffende bestandsinformatietabel.
        **Opmerking** Andere kenmerken dan de bestandsversie kunnen tijdens de installatie worden veranderd. Vergelijking van andere bestandskenmerken met de gegevens in de bestandsinformatietabel vormt geen ondersteunde methode om te controleren of de update is toegepast. Bovendien kunnen in bepaalde gevallen de namen van bestanden tijdens installatie worden gewijzigd. Als de bestands- of versiegegevens niet aanwezig zijn, gebruikt u een van de andere beschikbare methoden om te controleren of de update is geïnstalleerd.

-   **Verificatie van registersleutel**

    Aan de hand van de registersleutels in de **Referentietabel** in deze sectie kunt u ook controleren welke bestanden met deze beveiligingsupdate zijn geïnstalleerd.

    Deze registersleutels bevatten mogelijk geen volledige lijst met geïnstalleerde bestanden. Bovendien zijn deze registersleutels mogelijk niet correct als een beheerder of een OEM de beveiligingsupdate integreert in de bronbestanden van de Windows-installatie.
