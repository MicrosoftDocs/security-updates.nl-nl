Security Bulletin

Microsoft-beveiligingsbulletin MS11-064 - Belangrijk
====================================================

Beveiligingslekken in de TCP/IP-stack kunnen leiden tot denial of service (2563894)
-----------------------------------------------------------------------------------

Gepubliceerd: dinsdag 9 augustus 2011

**Versie:** 1.0

### Algemene informatie

#### Samenvatting

Met deze beveiligingsupdate worden twee privé gemelde beveiligingslekken in Microsoft Windows opgelost. De beveiligingslekken kunnen leiden tot denial of service als een aanvaller een reeks speciaal vervaardigde ICMP-berichten (Internet Control Message Protocol) verzendt naar een systeem of een speciaal vervaardigde URL-aanvraag verzendt naar een server voor webinhoud waarop de QoS-functie (Quality of Service) op URL-basis is ingeschakeld.

Het prioriteitsniveau van deze beveiligingsupdate is Belangrijk voor alle ondersteunde edities van Windows Server 2008 en Windows Server 2008 R2. Het prioriteitsniveau van deze beveiligingsupdate is Gemiddeld voor alle ondersteunde edities van Windows Vista en Windows 7. Zie de subsectie **Software waarin dit probleem optreedt en software waarin dit probleem niet optreedt** in deze sectie voor meer informatie.

De beveiligingsupdate lost de beveiligingslekken op doordat de manier wordt gewijzigd waarop de Windows TCP/IP-stack ICMP-berichten verwerkt en URL's in het geheugen verwerkt wanneer de QoS-functie op URL-basis is ingeschakeld. Zie de subsectie Veelgestelde vragen over het specifieke beveiligingslek in de volgende sectie, **Informatie over het beveiligingslek**, voor meer informatie over de beveiligingslekken.

**Aanbeveling.** De meeste klanten hebben automatische updates ingeschakeld en hoeven geen actie te ondernemen omdat deze beveiligingsupdate automatisch wordt gedownload en geïnstalleerd. Klanten die automatische updates niet hebben ingeschakeld, moeten controleren op updates en deze update handmatig installeren. Zie [Microsoft Knowledge Base-artikel 294871](http://support.microsoft.com/kb/294871) voor informatie over specifieke configuratieopties voor het uitvoeren van automatische updates.

Als het gaat om beheerders en bedrijfinstallaties of eindgebruikers die deze beveiligingsupdate handmatig willen installeren, adviseert Microsoft klanten de update zo snel mogelijk toe te passen met behulp van de software voor updatebeheer of door op updates te controleren met de service [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Zie ook de sectie **Hulpmiddelen en richtlijnen voor detecteren en implementeren**, verderop in dit bulletin.

Beveiligingslek met betrekking tot denial of service in ICMP - CVE-2011-1871  
----------------------------------------------------------------------------
  
<span></span>
Er bestaat een beveiligingslek met betrekking tot denial of service in de Windows TCP/IP-stack die wordt veroorzaakt wanneer de TCP/IP-stack een reeks speciaal vervaardigde ICMP-berichten onjuist verwerkt. Een aanvaller die dit lek weet te misbruiken, kan ervoor zorgen dat het systeem waarin dit probleem optreedt, niet meer reageert en automatisch opnieuw wordt opgestart.
  
Zie [CVE-2011-1871](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-1871) als u dit beveiligingslek wilt weergeven als standaardvermelding in de lijst met veelvoorkomende beveiligingslekken en blootstellingen.
  
#### Beperkende factoren voor het beveiligingslek met betrekking tot denial of service in ICMP - CVE-2011-1871
  
Microsoft heeft nog geen beperkende factoren voor dit beveiligingslek.
  
#### Tijdelijke oplossingen voor het beveiligingslek met betrekking tot denial of service in ICMP - CVE-2011-1871
  
Tijdelijke oplossingen zijn wijzigingen in een instelling of configuratie waardoor het beveiligingslek zelf niet wordt gedicht, maar waarmee aanvalsvectoren wel worden geblokkeerd voordat u de update uitvoert. Microsoft heeft de volgende methoden en statussen om het probleem te omzeilen getest om uit te zoeken of een tijdelijke oplossing functionaliteit vermindert:
  
-   **ICMP blokkeren bij de firewall**
  
    Door alle ICMP-berichten bij de firewall of router te blokkeren, zorgt u ervoor dat op systemen achter deze firewall of router geen aanvallen kunnen worden uitgevoerd. U wordt aanbevolen alle ongewenste inkomende communicatie via internet te blokkeren.
  
    **Gevolgen van de tijdelijke oplossing.** Deze tijdelijke oplossing kan een negatief effect hebben op de prestaties, omdat TCP de netwerkcommunicatie niet kan optimaliseren. ICMP-netwerkpakketten kunnen de fragmentatie van routers die netwerken van verschillende MTU's verbinden, uitschakelen. Fragmentatie verlaagt de TCP-doorvoer en verhoogt de belasting van het netwerk.
  
#### Veelgestelde vragen over het beveiligingslek met betrekking tot denial of service in ICMP - CVE-2011-1871
  
**Wat is de omvang van het beveiligingslek?**    
Dit beveiligingslek heeft betrekking op een denial of service.
  
**Waardoor wordt het beveiligingslek veroorzaakt?**    
Het beveiligingslek wordt veroorzaakt wanneer de Windows TCP/IP-stack een reeks speciaal vervaardigde ICMP-berichten onjuist verwerkt.
  
**In welke onderdeel treedt het beveiligingslek op?**    
Tcpip.sys is het onderdeel dat door het beveiligingslek wordt getroffen.
  
**Met welk doel kan een aanvaller het beveiligingslek misbruiken?**    
Een aanvaller die dit lek weet te misbruiken, kan ervoor zorgen dat het systeem waarin dit probleem optreedt, niet meer reageert en automatisch opnieuw wordt opgestart.
  
**Hoe kan een aanvaller dit beveiligingslek misbruiken?**    
Een externe, niet-geverifieerde aanvaller kan misbruik maken van dit beveiligingslek door een programma te ontwikkelen dat een reeks speciaal vervaardigde ICMP-berichten naar een systeem verzendt.
  
**Voor welke systemen vormt dit beveiligingslek het grootste risico?**    
Servers en werkstations worden door dit beveiligingslek getroffen.
  
**Wat doet de update?**    
De update lost het beveiligingslek op doordat de manier wordt gewijzigd waarop de Windows TCP/IP-stack ICMP-berichten verwerkt.
  
**Was dit beveiligingslek openbaar gemaakt op het moment dat dit beveiligingsbulletin werd uitgegeven?**    
Nee. Microsoft heeft uit betrouwbare bronnen informatie over dit beveiligingslek ontvangen.
  
**Was aan Microsoft gerapporteerd dat er misbruik was gemaakt van dit beveiligingslek op het moment dat dit beveiligingsbulletin werd uitgegeven?**    
Nee. Microsoft heeft geen informatie ontvangen die erop wees dat dit beveiligingslek was gebruikt om aanvallen uit te voeren op klanten op het moment dat dit beveiligingsbulletin oorspronkelijk werd uitgegeven.
  
Beveiligingslek met betrekking tot denial of service in TCP/IP QOS - CVE-2011-1965  
----------------------------------------------------------------------------------
  
<span></span>
Er bestaat een beveiligingslek met betrekking tot denial of service in de Windows TCP/IP-stack die wordt veroorzaakt wanneer de TCP/IP-stack URL's in het geheugen onjuist verwerkt wanneer Quality of Service (QoS) op URL-basis is ingeschakeld. Een aanvaller die dit lek weet te misbruiken, kan ervoor zorgen dat het systeem waarin dit probleem optreedt, niet meer reageert en automatisch opnieuw wordt opgestart.
  
Zie [CVE-2010-1965](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-1965) als u dit beveiligingslek wilt weergeven als standaardvermelding in de lijst met veelvoorkomende beveiligingslekken en blootstellingen.
  
#### Beperkende factoren voor het beveiligingslek met betrekking tot denial of service in TCP/IP QOS - CVE-2011-1965
  
Met beperkende factoren wordt verwezen naar een instelling, de algehele configuratie of algemene tips voor een standaardstatus waarmee de ernst van misbruik via een beveiligingslek kan worden verminderd. De volgende beperkende factoren kunnen in uw situatie van pas komen:
  
-   Standaard is de QoS-functie op URL-basis niet ingeschakeld op Windows-besturingssystemen. Gebruikers kunnen alleen worden getroffen door dit beveiligingslek als zij deze functie handmatig hebben geïnstalleerd.
  
#### Tijdelijke oplossingen voor het beveiligingslek met betrekking tot denial of service in TCP/IP QOS - CVE-2011-1965
  
Tijdelijke oplossingen zijn wijzigingen in een instelling of configuratie waardoor het beveiligingslek zelf niet wordt gedicht, maar waarmee aanvalsvectoren wel worden geblokkeerd voordat u de update uitvoert. Microsoft heeft de volgende methoden en statussen om het probleem te omzeilen getest om uit te zoeken of een tijdelijke oplossing functionaliteit vermindert:
  
-   QoS op beleidsbasis uitschakelen Zie het Microsoft TechNet-artikel [Policy-based Quality of Service (QoS)](http://technet.microsoft.com/en-us/library/dd919203(ws.10).aspx) voor meer informatie.
  
#### Veelgestelde vragen over het beveiligingslek met betrekking tot denial of service in TCP/IP QOS - CVE-2011-1965
  
**Wat is de omvang van het beveiligingslek?**    
Dit beveiligingslek heeft betrekking op een denial of service.
  
**Waardoor wordt het beveiligingslek veroorzaakt?**    
Het beveiligingslek wordt veroorzaakt wanneer de TCP/IP-stack speciaal vervaardigde URL's in het geheugen onjuist verwerkt. Dit beveiligingslek wordt alleen geactiveerd wanneer het systeem webinhoud host met een service als IIS en QoS op URL-basis is ingeschakeld.
  
**In welke onderdeel treedt het beveiligingslek op?**    
Tcpip.sys is het onderdeel dat door het beveiligingslek wordt getroffen.
  
**Wat is Quality of Service (QoS) op URL-basis?**    
Quality of Service (QoS) voor netwerken is een algemene set standaarden en mechanismen die ervoor zorgt dat kritieke toepassingen op hoog niveau presteren. Netwerkbeheerders kunnen QoS-mechanismen gebruiken om prioriteit te geven aan netwerkverkeer en de snelheid van dit verkeer te bepalen. Dankzij deze mechanismen worden de beschikbare middelen optimaal gebruikt voor het gewenste serviceniveau.
  
In Windows Server 2008 R2 kunnen IT-professionals prioriteit geven aan webverkeer op basis van de URL. Met QoS op URL-basis kunnen IT-professionals ervoor zorgen dat belangrijk webverkeer voorrang heeft op minder belangrijk verkeer, waardoor de prestaties in drukke netwerken verbeteren. IT-professionals kunnen bijvoorbeeld webverkeer voor kritieke interne websites een hogere prioriteit geven dan externe websites, zodat de prestaties optimaal zijn wanneer het netwerk zwaar wordt belast. Zie het Microsoft TechNet-artikel [URL-based QoS](http://technet.microsoft.com/en-us/library/dd637810(ws.10).aspx) voor meer informatie.
  
**Wat is a URI?**    
Een URI (Uniform Resource Identifier) is een tekenreeks die wordt gebruikt om iets te doen met internet- of netwerkbronnen of om die te identificeren. Een URL is een typisch voorbeeld van een URI die betrekking heeft op een bron, zoals een website. Zie [RFC-2396](http://www.ietf.org/rfc/rfc2396.txt) voor meer informatie over URI's.
  
**Met welk doel kan een aanvaller het beveiligingslek misbruiken?**    
Een aanvaller die dit lek weet te misbruiken, kan ervoor zorgen dat het systeem waarin dit probleem optreedt, niet meer reageert en automatisch opnieuw wordt opgestart.
  
**Hoe kan een aanvaller dit beveiligingslek misbruiken?**    
In het geval van een externe aanval kan een niet-geverifieerde aanvaller misbruik maken van dit beveiligingslek door een speciaal vervaardigde URL-aanvraag te verzenden naar een server voor webinhoud waarop QoS op URL-basis is ingeschakeld.
  
**Voor welke systemen vormt dit beveiligingslek het grootste risico?**    
Dit beveiligingslek vormt het grootste risico voor servers voor webinhoud waarop QoS op URL-basis is ingeschakeld. Servers waarop QoS op URL-basis niet is ingeschakeld of die niet bestemd zijn voor webinhoud, worden niet getroffen. Werkstations zoals Windows 7-systemen, waarop IIS (Internet Information Services) kan worden uitgevoerd, lopen meer risico als zowel IIS als QoS op URL-basis is ingeschakeld.
  
**Lopen Windows 7-systemen meer risico als deze zijn geconfigureerd als server met IIS?**    
Ja. Serversoftware zoals IIS kan worden geïnstalleerd en geconfigureerd op Windows 7-systemen. Hoewel Windows 7-systemen worden beschouwd als werkstations, werken zij in deze configuratie als servers en daarom lopen ze meer risico.
  
**Wat doet de update?**    
De update lost het beveiligingslek op doordat de manier wordt gewijzigd waarop de TCP/IP-stack URL's in het geheugen verwerkt wanneer QoS op URL-basis is ingeschakeld.
  
**Was dit beveiligingslek openbaar gemaakt op het moment dat dit beveiligingsbulletin werd uitgegeven?**    
Nee. Microsoft heeft uit betrouwbare bronnen informatie over dit beveiligingslek ontvangen.
  
**Was aan Microsoft gerapporteerd dat er misbruik was gemaakt van dit beveiligingslek op het moment dat dit beveiligingsbulletin werd uitgegeven?**    
Nee. Microsoft heeft geen informatie ontvangen die erop wees dat dit beveiligingslek was gebruikt om aanvallen uit te voeren op klanten op het moment dat dit beveiligingsbulletin oorspronkelijk werd uitgegeven.
