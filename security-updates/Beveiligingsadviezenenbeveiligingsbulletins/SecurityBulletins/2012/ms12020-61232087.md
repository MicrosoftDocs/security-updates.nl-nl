Security Bulletin

Microsoft-beveiligingsbulletin MS12-020 - Kritiek
=================================================

Door beveiligingslekken in Remote Desktop kan externe code worden uitgevoerd (2671387)
--------------------------------------------------------------------------------------

Gepubliceerd: dinsdag 13 maart 2012 | Bijgewerkt: dinsdag 12 juni 2012

**Versie:** 2.0

### Algemene informatie

#### Samenvatting

Met deze beveiligingsupdate worden twee privé gemeld beveiligingslekken in Remote Desktop Protocol opgelost. Het ernstigste beveiligingslek kan leiden tot uitvoering van externe code als een aanvaller een reeks speciaal vervaardigde RDP-pakketten naar een getroffen systeem verzendt. Standaard is Remote Desktop Protocol (RDP) niet ingeschakeld op een Windows-besturingssysteem. Systemen waarop RDP niet is ingeschakeld, lopen geen risico.

Het prioriteitsniveau van deze beveiligingsupdate is Kritiek voor alle ondersteunde versies van Microsoft Windows. Zie de subsectie **Software waarin dit probleem optreedt en software waarin dit probleem niet optreedt** in deze sectie voor meer informatie.

De beveiligingsupdate lost de beveiligingslekken op doordat de manier wordt gewijzigd waarop Remote Desktop Protocol pakketten in het geheugen verwerkt en de manier waarop de RDP-service pakketten verwerkt. Zie de subsectie Veelgestelde vragen over het specifieke beveiligingslek in de volgende sectie, **Informatie over het beveiligingslek**, voor meer informatie over de beveiligingslekken.

**Aanbeveling.** De meeste klanten hebben automatische updates ingeschakeld en hoeven geen actie te ondernemen omdat deze beveiligingsupdate automatisch wordt gedownload en geïnstalleerd. Klanten die automatische updates niet hebben ingeschakeld, moeten controleren op updates en deze update handmatig installeren. Zie [Microsoft Knowledge Base-artikel 294871](http://support.microsoft.com/kb/294871) voor informatie over specifieke configuratieopties voor het uitvoeren van automatische updates.

Als het gaat om beheerders en bedrijfinstallaties of eindgebruikers die deze beveiligingsupdate handmatig willen installeren, adviseert Microsoft klanten de update onmiddellijk toe te passen met behulp van de software voor updatebeheer of door op updates te controleren met de service [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Zie ook de sectie **Hulpmiddelen en richtlijnen voor detecteren en implementeren**, verderop in dit bulletin.

**Bekende problemen.** In [Microsoft Knowledge Base-artikel 2671387](http://support.microsoft.com/kb/2671387) worden bekende problemen beschreven die klanten mogelijk ondervinden bij de installatie van deze beveiligingsupdate. In het artikel worden ook de aanbevolen oplossingen voor deze problemen toegelicht.

#### Software waarin dit probleem optreedt en software waarin dit probleem niet optreedt

De volgende software is getest om te bepalen in welke versies of edities dit probleem optreedt. Andere versies of edities hebben het einde van hun ondersteuningscyclus bereikt of hebben geen last van dit beveiligingslek. Ga naar [Microsoft Support Lifecycle](http://go.microsoft.com/fwlink/?linkid=21742) om de ondersteuningscyclus voor uw product en versie te bepalen.

Veelgestelde vragen met betrekking tot deze beveiligingsupdate
--------------------------------------------------------------

<span></span>
**Waarom is dit bulletin opnieuw uitgebracht op 12 juni 2012?** **
**Microsoft heeft wijzigingen aangebracht in dit bulletin om beveiligingsupdate KB2667402 opnieuw aan te bieden voor alle edities van Windows 7 en Windows Server 2008 R2. Door de wijzigingen in deze update wordt een probleem opgelost in de update die oorspronkelijk is aangeboden op 13 maart 2012, waarbij de update wordt geïnstalleerd op Windows 7 of Windows Server 2008 R2 en het besturingssysteem vervolgens wordt bijgewerkt naar Service Pack 1. Dit scenario heeft als gevolg dat de verkeerde binaire bestandversie van rdpcorekmts.dll (6.1.7600.16952) wordt geïnstalleerd in plaats van de juiste versie (6.1.7601.17514). Systemen met ondersteunde edities van Windows 7 en Windows Server 2008 R2 krijgen de nieuwe versie van deze update automatisch aangeboden. Klanten met deze besturingssystemen moeten de nieuwe versie van de update installeren, ook als zij de oorspronkelijke update van 13 maart 2012 hebben geïnstalleerd.

**Waar is de informatie over de bestandsgegevens?**
Zie de referentietabellen in de sectie **Implementatie van de beveiligingsupdate** voor de details van de bestandsgegevens.

**Wordt Windows** **DeveloperPreview getroffen door** **de beveiligingslekken** **die worden beschreven in dit bulletin?**
Ja. De KB2621440-update is beschikbaar voor Windows Developer Preview. Klanten met Windows Developer Preview worden aangeraden de update toe te passen op hun systemen. De update is alleen verkrijgbaar via [Windows Update](http://go.microsoft.com/fwlink/?linkid=21130).

**Ik gebruik een getroffen besturingssysteem maar ik heb** **Remote Desktop Protocol (RDP)** **niet ingeschakeld. Krijg ik deze update toch aangeboden?**
Ja. Hoewel systemen waarop RDP niet is ingeschakeld geen risico lopen op de beveiligingslekken, krijgen klanten die RDP niet hebben ingeschakeld deze beveiligingsupdate toch aangeboden ter bescherming van hun systeem.

**Er worden twee updates vermeld voor de** **versie van Microsoft Windows** **die is geïnstalleerd op mijn systeem. Moet ik beide updates installeren?**
Ja. Klanten moeten alle updates toepassen die worden aangeboden voor de versie van Microsoft Windows die is geïnstalleerd op hun systeem.

**Moet ik deze beveiligingsupdates in een bepaalde volgorde installeren?**
Nee. Meerdere updates voor één getroffen systeem kunnen in willekeurige volgorde worden toegepast.

**Ik gebruik een oudere versie van de software die staat beschreven in dit beveiligingsbulletin. Wat moet ik doen?**
De software waarin het probleem optreedt en die in dit beveiligingsbulletin wordt vermeld, is getest om te controleren of het probleem bij deze versies optreedt. Andere versies hebben het einde van hun ondersteuningscyclus bereikt. Ga naar de volgende [Microsoft Support Lifecycle](http://go.microsoft.com/fwlink/?linkid=21742)-website voor meer informatie over de levenscyclus van Windows-producten.

Klanten die met deze besturingssysteemversies werken, kunnen er het beste voor kiezen om binnenkort te migreren naar versies waarvoor wel ondersteuning wordt aangeboden om te voorkomen dat potentiële beveiligingslekken een probleem vormen. Zie [Selecteer een product voor lifecycle-informatie](http://go.microsoft.com/fwlink/?linkid=169555) om de ondersteuningscyclus voor uw softwareversie te bepalen. Zie [Service Pack Lifecycle Support Policy](http://go.microsoft.com/fwlink/?linkid=89213) voor meer informatie over de servicepacks voor deze softwareversies.

Klanten die aangepaste ondersteuning voor deze producten nodig hebben, dienen voor aangepaste ondersteuningsopties contact op te nemen met hun Microsoft-accountteamvertegenwoordiger, hun technische accountmanager of de aangewezen vertegenwoordiger van hun Microsoft-partner. Klanten zonder Alliance-, Premier- of Authorized-contract kunnen contact opnemen met het Microsoft-verkoopkantoor in hun land. Voor contactinformatie gaat u naar de website [Microsoft Worldwide](http://go.microsoft.com/fwlink/?linkid=33329) en selecteert u het land in de lijst Contact Information. Klik vervolgens op **Go** om een lijst met telefoonnummers weer te geven. Wanneer u belt, vraag dan naar de Premier Support-verkoopmanager. Zie de [Veelgestelde vragen over het Microsoft Support Lifecycle-beleid](http://go.microsoft.com/fwlink/?linkid=169557) voor meer informatie.

Beveiligingslek in Remote Desktop Protocol - CVE-2012-0002  
----------------------------------------------------------
  
<span></span>
Er bestaat een beveiligingslek met betrekking tot het uitvoeren van externe code door de manier waarop Remote Desktop Protocol toegang krijgt tot een object in het geheugen dat onjuist is geïnitialiseerd of dat is verwijderd. Een aanvaller die het beveiligingslek misbruikt, kan willekeurige code op het doelsysteem uitvoeren. De aanvaller kan vervolgens programma's installeren, gegevens bekijken, wijzigen of wissen, of nieuwe accounts met volledige gebruikersrechten maken.
  
Zie [CVE-2012-0002](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2012-0002) als u dit beveiligingslek wilt weergeven als standaardvermelding in de lijst met veelvoorkomende beveiligingslekken en blootstellingen.
  
#### Beperkende factoren voor het beveiligingslek in Remote Desktop Protocol - CVE-2012-0002
  
Met beperkende factoren wordt verwezen naar een instelling, de algehele configuratie of algemene tips voor een standaardstatus waarmee de ernst van misbruik via een beveiligingslek kan worden verminderd. De volgende beperkende factoren kunnen in uw situatie van pas komen:
  
-   Standaard is Remote Desktop Protocol niet ingeschakeld op een Windows-besturingssysteem. Systemen waarop RDP niet is ingeschakeld, lopen geen risico. Let op: op Windows XP en Windows Server 2003 kan de functie Hulp op afstand het RDP-protocol inschakelen.  
-   Met aanbevolen procedures voor firewalls en standaardfirewallconfiguraties kunt u netwerken beveiligen tegen aanvallen die van buiten het bedrijfsnetwerk komen. U wordt aanbevolen op de systemen die zijn rechtstreeks toegang hebben tot internet zo min mogelijk poorten bloot te stellen aan deze aanvallen.
  
#### Tijdelijke oplossingen voor het beveiligingslek in Remote Desktop Protocol - CVE-2012-0002
  
Tijdelijke oplossingen zijn wijzigingen in een instelling of configuratie waardoor het beveiligingslek zelf niet wordt gedicht, maar waarmee aanvalsvectoren wel worden geblokkeerd voordat u de update uitvoert. Microsoft heeft de volgende methoden en statussen om het probleem te omzeilen getest om uit te zoeken of een tijdelijke oplossing functionaliteit vermindert:
  
-   **Terminal Services, Extern bureaublad, Hulp op afstand, en Externe Internet-werkplek (in Windows Small Business Server 2003) uitschakelen als u deze functies niet langer nodig hebt**
  
    Als u deze services niet langer nodig hebt, is het een goed idee deze uit te schakelen. Hierdoor bent u beter beveiligd tegen beveiligingslekken.
  
    Zie [To disable Remote Desktop](http://www.microsoft.com/resources/documentation/windows/xp/all/proddocs/en-us/rdesktop_disable.mspx) voor informatie over het handmatig uitschakelen van Extern bureaublad.
  
    Zie [Microsoft Knowledge Base-artikel 306300](http://support.microsoft.com/kb/306300) voor informatie over het uitschakelen van Extern bureaublad met behulp van Groepsbeleid.
  
    Zie het TechNet-artikel [Remote Assistance](http://technet.microsoft.com/en-us/library/bb457177.aspx) voor informatie over Hulp op afstand en het handmatig en via Groepsbeleid uitschakelen van deze functie.
  
    Zie het TechNet-artikel [Securing Your Windows Small Business Server 2003 Network](http://www.microsoft.com/technet/security/secnews/articles/sec_sbs2003_network.mspx) voor informatie over het uitschakelen van de functies Terminal Services en Externe Internet-werkplek in Windows Small Business Server 2003.
  
-   **TCP-poort 3389 blokkeren op de bedrijfsfirewall**
  
    TCP-poort 3389 wordt gebruikt om een verbinding tot stand te brengen met het onderdeel waarin dit probleem optreedt. Door deze poort op de firewall te blokkeren, zorgt u ervoor dat op systemen achter deze firewall geen aanvallen kunnen worden uitgevoerd. Zo kunt u netwerken beveiligen tegen aanvallen die van buiten het bedrijfsnetwerk komen. Het blokkeren van de getroffen poorten op de firewall is de beste verdediging tegen internetaanvallen. Systemen kunnen echter nog steeds van binnen het bedrijf worden aangevallen.
  
    Op Windows XP en Windows Server 2003 kan de Windows Firewall helpen bij de beveiliging van individuele computers. Windows Firewall staat standaard geen verbindingen toe via deze poort, behalve in Windows XP Service Pack 2 als de functie Extern bureaublad is ingeschakeld. Zie het TechNet-artikel [Enable or Disable the Remote Desktop Firewall Rule](http://technet.microsoft.com/en-us/library/cc736451(ws.10).aspx) voor informatie over het uitschakelen van de uitzondering voor Extern bureaublad in Windows Firewall op deze platforms. Als u de uitzondering voor de functie Extern bureaublad niet kunt uitschakelen in Windows Firewall, kunt u wellicht de gevolgen van het beveiligingslek beperken door de standaardwaarde in te stellen op de optie voor het lokale netwerk in plaats van op de optie voor alle computers (inclusief computers op internet). Hierdoor verlaagt u de kans op aanvallen via internet.
  
    **Opmerking** In Windows Small Business Server 2003 zit een functie genaamd Externe Internet-werkplek. Deze functie gebruikt TCP-poort 4125 voor het verwerken van RDP-verbindingen. Als u deze functie gebruikt, dient u te controleren dat behalve poort 3389 ook deze poort is geblokkeerd voor internet.
  
    **Opmerking** U hebt wellicht andere poorten ingesteld voor de getroffen onderdelen. Als u dat hebt gedaan, moet u ook deze poorten blokkeren.
  
-   **Verificatie op netwerkniveau inschakelen op systemen met ondersteunde edities van Windows Vista, Windows 7, Windows Server 2008 en Windows Server 2008 R2**
  
    **  
    Opmerking** Zie [Microsoft Knowledge Base-artikel 2671387](http://support.microsoft.com/kb/2671387) als u de automatische **Microsoft Fix it**-oplossing wilt gebruiken om deze tijdelijke oplossing in te schakelen.
  
    U kunt verificatie op netwerkniveau inschakelen om te voorkomen dat niet-geverifieerde aanvallers misbruik maken van dit beveiligingslek. Als verificatie op netwerkniveau is ingeschakeld, moet een aanvaller zich eerst met een geldig account op het doelsysteem verifiëren bij Extern bureaublad-services voordat de aanvaller misbruik kan maken van het beveiligingslek.
  
    Uw omgeving moet voldoen aan de volgende vereisten als u verificatie op netwerkniveau wilt kunnen gebruiken:
  
    -   Op de clientcomputer moet ten minste Remote Desktop Connection 6.0 worden gebruikt.  
    -   Op de clientcomputer moet een besturingssysteem worden gebruikt dat het CredSSP-protocol (Credential Security Support Provider) ondersteunt, zoals Windows 7 of Windows Vista.  
    -   Op de hostserver voor Extern bureaublad-sessies moet Windows Server 2008 R2 of Windows Server 2008 worden uitgevoerd.
  
    Voer de volgende stappen uit om verificatie op netwerkniveau te configureren voor een verbinding:
  
    1.  Open het configuratieprogramma voor de Hostserver voor Extern bureaublad-sessies op de hostserver voor Extern bureaublad-sessies. U kunt het configuratieprogramma voor de Hostserver voor Extern bureaublad-sessies als volgt openen: klik op **Start**, wijs **Systee**m**beheer** aan, wijs **Extern bureaublad-services** aan en klik vervolgens op **Configuratieprogramma voor de Hostserver voor Extern bureaublad-sessies**.  
    2.  Klik onder **Verbindingen** met de rechtermuisknop op de naam van de verbinding en klik dan op **Eigenschappen**.  
    3.  Schakel op het tabblad **Algemeen** het selectievakje **Alleen verbindingen toestaan met computers met Extern bureaublad met verificatie op netwerkniveau** in.  
        Als het selectievakje **Alleen verbindingen toestaan met computers met Extern bureaublad met verificatie op netwerkniveau** is ingeschakeld en niet kan worden uitgeschakeld, is de groepsbeleidinstelling **Gebruikersverificatie met verificatie op netwerkniveau vereisen voor externe verbindingen** ingeschakeld en toegepast op de hostserver voor Extern bureaublad-sessies.  
    4.  Klik op **OK**.
  
    **Gevolgen van de tijdelijke oplossing.** Clientcomputers die het CredSSP-protocol (Credential Security Support Provider) niet ondersteunen, hebben geen toegang tot servers die zijn beveiligd met verificatie op netwerkniveau. **Opmerking** Als u een beheerder bent en u deze tijdelijke oplossing toepast in een gemengde netwerkomgeving waarbij systemen met Windows XP Service Pack 3 gebruik moeten maken van Extern bureaublad, raadpleegt u [Microsoft Knowledge Base-artikel 951608](http://support.microsoft.com/kb/951608) voor informatie over hoe u CredSSP kunt inschakelen in Windows XP Service Pack 3.
  
    Zie het TechNet-artikel [Configure Network Level Authentication for Remote Desktop Services Connections](http://technet.microsoft.com/en-us/library/cc732713.aspx) voor meer informatie over verificatie op netwerkniveau, waaronder informatie over hoe u verificatie op netwerkniveau kunt inschakelen met behulp van Groepsbeleid.
  
#### Veelgestelde vragen over het beveiligingslek in Remote Desktop Protocol - CVE-2012-0002
  
**Wat is de omvang van het beveiligingslek?**  
Dit beveiligingslek heeft betrekking op de uitvoering van externe code.
  
**Waardoor wordt het beveiligingslek veroorzaakt?**  
Het beveiligingslek wordt veroorzaakt wanneer Remote Desktop Protocol een reeks speciaal vervaardigde pakketten verwerkt waardoor toegang wordt verkregen tot een object in het geheugen dat niet goed is geïnitialiseerd of dat is verwijderd.
  
**Wat is Remote Desktop Protocol (RDP)?**  
Met Remote Desktop Protocol (RDP) kunnen gebruikers een virtuele sessie op hun computer opzetten. Op deze manier hebben gebruikers via een andere computer toegang tot al hun gegevens en toepassingen op hun eigen computer. Zie het MSDN-artikel [Remote Desktop Protocol](http://msdn.microsoft.com/en-us/library/aa383015(v=vs.85).aspx) voor meer informatie.
  
**Is RDP standaard ingeschakeld?**  
Nee, RDP is niet standaard ingeschakeld. Klanten die RDP niet hebben ingeschakeld, krijgen de update toch aangeboden zodat hun systemen goed zijn beschermd. Zie het TechNet-artikel [How to enable and to configure Remote Desktop for Administration in Windows Server 2003](http://support.microsoft.com/kb/814590) voor meer informatie over deze configuratie-instelling.
  
**Wat is verificatie op netwerkniveau voor Extern bureaublad-services?  
**Verificatie op netwerkniveau is een verificatiemethode die kan worden gebruikt om de beveiliging van de hostserver voor Extern bureaublad-sessies te verbeteren doordat de gebruiker zich moet aanmelden bij de hostserver voor Extern bureaublad-sessies voordat een sessie tot stand wordt gebracht. Zie het TechNet-artikel, [Configure Network Level Authentication for Remote Desktop Services Connections](http://technet.microsoft.com/en-us/library/cc732713.aspx) voor meer informatie over verificatie op netwerkniveau.
  
**Met welk doel kan een aanvaller het beveiligingslek misbruiken?**  
Een aanvaller die misbruik weet te maken van dit beveiligingslek, kan volledige controle krijgen over het systeem waarop dit probleem voorkomt. De aanvaller kan vervolgens programma's installeren, gegevens bekijken, wijzigen of wissen, of nieuwe accounts met volledige gebruikersrechten maken.
  
**Hoe kan een aanvaller dit beveiligingslek** **misbruiken?**  
Op systemen met ondersteunde edities van Windows XP en Windows Server 2003 kan een externe, niet-geverifieerde aanvaller misbruik maken van dit beveiligingslek door een reeks speciaal vervaardigde RDP-pakketten naar het doelsysteem te verzenden.
  
Op systemen met ondersteunde edities van Windows Vista, Windows 7, Windows Server 2008 en Windows Server 2008 R2 met verificatie op netwerkniveau kan een externe, niet-geverifieerde aanvaller misbruik maken van dit beveiligingslek door een reeks van speciaal vervaardigde RDP-pakketten naar het doelsysteem te verzenden.
  
Op systemen met ondersteunde edities van Windows Vista, Windows 7, Windows Server 2008 en Windows Server 2008 R2 waarop verificatie op netwerkniveau is ingeschakeld, moet een aanvaller zich eerst met een geldig account op het doelsysteem verifiëren bij Extern bureaublad-services. De aanvaller kan vervolgens misbruik maken van dit beveiligingslek door een reeks speciaal vervaardigde RDP-pakketten naar het doelsysteem te verzenden.
  
**Voor welke systemen vormt dit beveiligingslek het grootste risico?**  
Dit beveiligingslek vormt het grootste risico voor terminalservers. Systemen waarop RDP niet is ingeschakeld, lopen geen risico. Daarnaast lopen systemen met ondersteunde edities van Windows Vista, Windows 7, Windows Server 2008 en Windows Server 2008 R2 waarop verificatie op netwerkniveau is ingeschakeld, minder risico omdat een aanvaller zich eerst moet aanmelden bij de service.
  
**Wat doet de update?**  
De beveiligingsupdate lost het beveiligingslek op door de manier te wijzigen waarop Remote Desktop Protocol pakketten in het geheugen verwerkt.
  
**Was dit beveiligingslek openbaar gemaakt op het moment dat dit beveiligingsbulletin werd uitgegeven?**  
Nee. Microsoft heeft uit betrouwbare bronnen informatie over dit beveiligingslek ontvangen.
  
**Was aan Microsoft gerapporteerd dat er misbruik was gemaakt van dit beveiligingslek op het moment dat dit beveiligingsbulletin werd uitgegeven?**  
Nee. Microsoft heeft geen informatie ontvangen die erop wees dat dit beveiligingslek was gebruikt om aanvallen uit te voeren op klanten op het moment dat dit beveiligingsbulletin oorspronkelijk werd uitgegeven.

    Omdat er verschillende versies zijn van Microsoft Windows, kunnen de volgende stappen voor uw computer enigszins afwijken. Als dit het geval is, raadpleeg dan de productdocumentatie om deze stappen uit te voeren.

    1.  Klik op **Start** en vervolgens op **Zoeken**.
    2.  Klik in het deelvenster **Zoekresultaten** op **Alle bestanden en mappen** onder **Zoekassistent**.
    3.  Typ in het vak **De volledige of gedeeltelijke bestandsnaam** een bestandsnaam uit de betreffende bestandsinformatietabel en klik vervolgens op **Zoeken**.

        **Opmerking** Andere kenmerken dan de bestandsversie kunnen tijdens de installatie worden veranderd. Vergelijking van andere bestandskenmerken met de gegevens in de bestandsinformatietabel vormt geen ondersteunde methode om te controleren of de update is toegepast. Bovendien kunnen in bepaalde gevallen de namen van bestanden tijdens installatie worden gewijzigd.     Deze registersleutels bevatten mogelijk geen volledige lijst met geïnstalleerde bestanden. Bovendien zijn deze registersleutels mogelijk niet correct als een beheerder of een OEM de beveiligingsupdate integreert in de bronbestanden van de Windows-installatie.
