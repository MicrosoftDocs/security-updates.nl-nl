Security Bulletin

Microsoft-beveiligingsbulletin MS10-024 - Belangrijk
====================================================

Beveiligingslekken in Microsoft Exchange en de Windows SMTP-service kunnen leiden tot denial of service (981832)
----------------------------------------------------------------------------------------------------------------

Gepubliceerd: dinsdag 13 april 2010 | Bijgewerkt: dinsdag 13 juli 2010

**Versie:** 2.0

### Algemene informatie

#### Samenvatting

Met deze beveiligingsupdate wordt een openbaar gemaakt beveiligingslek en een privé gemeld beveiligingslek in Microsoft Exchange en Windows SMTP-service opgelost. De ernstigere beveiligingslekken kunnen leiden tot een denial of service als een aanvaller een speciaal vervaardigd DNS-antwoord verzendt naar een computer waarop de SMTP-service wordt uitgevoerd. Het SMTP-onderdeel wordt standaard niet geïnstalleerd op Windows Server 2003, Windows Server 2003 x64 Edition of Windows XP Professional x64 Edition.

Het prioriteitsniveau van deze beveiligingsupdate is Belangrijk voor alle ondersteunde edities van Microsoft Windows 2000, Windows XP en Windows Server 2003; 32-bits en x64-edities van Windows Server 2008; Windows Server 2008 R2 voor x64-systemen; en Microsoft Exchange Server 2003. Het prioriteitsniveau van deze beveiligingsupdate is Gemiddeld voor Microsoft Exchange Server 2000. Zie de subsectie **Software waarin dit probleem optreedt en software waarin dit probleem niet optreedt** in deze sectie voor meer informatie.

De beveiligingsupdate lost de beveiligingslekken op doordat de manier wordt verbeterd waarop SMTP MX-records parseert en de manier waarop SMTP geheugen toewijst voor het interpreteren van reacties op SMTP-opdrachten. Zie de subsectie Veelgestelde vragen over het specifieke beveiligingslek in de volgende sectie, **Informatie over het beveiligingslek**, voor meer informatie over de beveiligingslekken.

**Aanbeveling.** De meeste klanten hebben automatische updates ingeschakeld en hoeven geen actie te ondernemen omdat deze beveiligingsupdate automatisch wordt gedownload en geïnstalleerd. Klanten die automatische updates niet hebben ingeschakeld, moeten controleren op updates en deze update handmatig installeren. Zie [Microsoft Knowledge Base-artikel 294871](http://support.microsoft.com/kb/294871) voor informatie over specifieke configuratieopties voor het uitvoeren van automatische updates.

Als het gaat om beheerders en bedrijfinstallaties of eindgebruikers die deze beveiligingsupdate handmatig willen installeren, adviseert Microsoft klanten de update zo snel mogelijk toe te passen met behulp van de software voor updatebeheer of door op updates te controleren met de service [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Zie ook de sectie **Hulpmiddelen en richtlijnen voor detecteren en implementeren**, verderop in dit bulletin.

**Bekende problemen.** In [Microsoft Knowledge Base-artikel 981832](http://support.microsoft.com/kb/981832) worden bekende problemen beschreven die klanten mogelijk ondervinden bij de installatie van deze beveiligingsupdate. In het artikel worden ook de aanbevolen oplossingen voor deze problemen toegelicht.

Andere versies of edities hebben het einde van hun ondersteuningscyclus bereikt of hebben geen last van dit beveiligingslek. Ga naar [Microsoft Support Lifecycle](http://go.microsoft.com/fwlink/?linkid=21742) om de ondersteuningscyclus voor uw product en versie te bepalen.

Veelgestelde vragen met betrekking tot deze beveiligingsupdate
--------------------------------------------------------------

<span></span>
**Waarom is dit bulletin op 13 juli 2010 opnieuw uitgebracht?**  
Microsoft heeft dit bulletin opnieuw uitgebracht om de update voor Windows Server 2008 en Windows Server 2008 R2 opnieuw aan te bieden en daarmee een bekend probleem dat wordt beschreven in KB976323 op te lossen. De opties voor SMTP-configuraties worden opnieuw ingesteld nadat u deze beveiligingsupdate installeert op een systeem met Windows Server 2008 waarop Internet Information Services (IIS) is geïnstalleerd. Opties voor SMTP-configuraties die u hebt ingesteld voordat u de update installeert, gaan verloren. Benodigde opties voor SMTP-configuraties moeten handmatig opnieuw worden ingesteld nadat u deze beveiligingsupdate hebt geïnstalleerd.

De opnieuw aangeboden beveiligingsupdate is alleen van toepassing op de pakketten die worden beschreven in KB976323. Klanten die hun systemen al hebben bijgewerkt, hoeven deze update niet opnieuw te installeren.

**Waarom is dit beveiligingsbulletin op 15 april 2010 bijgewerkt?**  
Microsoft heeft dit beveiligingsbulletin bijgewerkt met informatie over een detectiewijziging voor Microsoft Exchange Server 2000 Service Pack 3. Dit is alleen een detectiewijziging. Door de verbeterde detectie wordt de beveiligingsupdate nu ook aangeboden aan getroffen systemen waaraan deze beveiligingsupdate voorheen niet werd aangeboden. De beveiligingsupdates in dit bulletin zijn niet gewijzigd. Klanten die update KB976703 al op hun systeem hebben geïnstalleerd, hoeven deze update niet opnieuw te installeren.

**Waar is de informatie over de bestandsgegevens?**  
Zie de referentietabellen in de sectie **Implementatie van de beveiligingsupdate** voor de details van de bestandsgegevens.

**In welke opzichten verschillen de updates voor Microsoft Exchange Server 2007 en Microsoft Exchange Server 2010 van die voor Microsoft Exchange Server 2000 en Microsoft Exchange Server 2003?**  
Met de release van Microsoft Exchange 2007 begeeft Microsoft Exchange zich naar een nieuw servicemodel. Meer informatie over het servicemodel van Microsoft Exchange vindt u in het TechNet-artikel [Exchange 2007 Servicing](http://go.microsoft.com/fwlink/?linkid=89577). Vragen over het nieuwe servicemodel van Microsoft Exchange kunt u richten aan [Help en ondersteuning van Microsoft](http://support.microsoft.com/).

**Op welke manier verhelpt deze update een aantal gerapporteerde beveiligingslekken?**  
Deze update biedt hulp bij diverse beveiligingslekken omdat een aantal wijzigingen dat nodig is om de gesignaleerde problemen op te lossen, in de bijbehorende bestanden is opgenomen. Klanten hoeven op deze manier niet allerlei gelijksoortige updates te installeren, maar kunnen volstaan met deze ene update.

**Bevat deze update andere functionaliteitswijzigingen met betrekking tot beveiliging?**  
Ja. Naast de wijzigingen die in de sectie Informatie over het beveiligingslek van dit bulletin worden beschreven, bevat deze update een ingrijpende wijziging voor Microsoft Exchange 2000 Server Service Pack 3 om een probleem met de Chinese tekenset bij gebruik van Outlook Web Access (OWA) op te lossen. Dit beveiligingslek heeft alleen gevolgen voor Exchange 2000-systemen. Let op: voor de beveiligingsupdate voor Microsoft Exchange 2000 Server Service Pack 3 is het updatepakket voor Exchange 2000 Server Post-Service Pack 3 (SP3) vereist. U dient het updatepakket voor Exchange 2000 (KB870540) te installeren alvorens u de ingrijpende beveiligingsupdate installeert die bij dit beveiligingsbulletin wordt geleverd. Deze update detecteert of het updatepakket is geïnstalleerd. Indien het updatepakket niet is geïnstalleerd, wordt u verwezen naar de website waarop de download is te vinden. Zie [Microsoft Knowledge Base-artikel 870540](http://support.microsoft.com/kb/870540) voor meer informatie. Zie [Updatepakket voor Exchange 2000 (KB870540)](http://www.microsoft.com/downloads/details.aspx?familyid=363a57a4-8bed-4bbb-bbe4-abc11ab04611) om de vereiste update te downloaden.

Deze update bevat ook een ingrijpende wijziging voor Microsoft Exchange 2007 en Microsoft Exchange 2010, die extra bronpoortentropie toevoegt aan DNS-transacties die worden gestart door de SMTP-service.

**Moet ik updates voor zowel Windows als Exchange toepassen?**  
Op systemen waarop Microsoft Exchange is geïnstalleerd, moet zowel de Exchange- als de Windows-update worden toegepast. Als u de SMTP-service hebt ingeschakeld maar u de Exchange-service niet uitvoert, hoeft u alleen de Windows-update toe te passen.

**Ik gebruik Exchange 2007 of Exchange 2010. Waarom krijg ik een update aangeboden als deze software niet wordt getroffen door de beveiligingslekken die in dit bulletin worden beschreven?**  
De updates voor Microsoft Exchange 2007 en Microsoft Exchange 2010 bevatten alleen de ingrijpende wijziging die extra bronpoortentropie toevoegt aan DNS-transacties die worden gestart door de SMTP-service.

**Moet ik het updatepakket voor Exchange Server 2007-servers in een bepaalde volgorde installeren?**  
Raadpleeg het TechNet-artikel over [het installeren van het nieuwste service pack of updatepakket voor Exchange 2007](http://technet.microsoft.com/en-us/library/ee221147(exchg.80).aspx).

**Ik gebruik een oudere versie van de software die staat beschreven in dit beveiligingsbulletin. Wat moet ik doen?**  
De software waarin het probleem optreedt en die in dit beveiligingsbulletin wordt vermeld, is getest om te controleren of het probleem bij deze versies optreedt. Andere versies hebben het einde van hun ondersteuningscyclus bereikt. Ga naar de volgende [Microsoft Support Lifecycle](http://go.microsoft.com/fwlink/?linkid=21742)-website voor meer informatie over de levenscyclus van Windows-producten.

Klanten die met deze besturingssysteemversies werken, kunnen er het beste voor kiezen om binnenkort te migreren naar versies waarvoor wel ondersteuning wordt aangeboden om te voorkomen dat potentiële beveiligingslekken een probleem vormen. Zie [Selecteer een product voor lifecycle-informatie](http://go.microsoft.com/fwlink/?linkid=169555) om de ondersteuningscyclus voor uw softwareversie te bepalen. Zie [Lifecycle ondersteunde service packs](http://go.microsoft.com/fwlink/?linkid=89213) voor meer informatie over service packs voor deze softwareversies.

Klanten die aangepaste ondersteuning voor deze producten nodig hebben, dienen voor aangepaste ondersteuningsopties contact op te nemen met hun Microsoft-accountteamvertegenwoordiger, hun technische accountmanager of de aangewezen vertegenwoordiger van hun Microsoft-partner. Klanten zonder Alliance-, Premier- of Authorized-contract kunnen contact opnemen met het Microsoft-verkoopkantoor in hun land. Voor contactinformatie gaat u naar de website [Microsoft Worldwide](http://go.microsoft.com/fwlink/?linkid=33329) en selecteert u het land in de lijst Contact Information. Klik vervolgens op **Go** om een lijst met telefoonnummers weer te geven. Wanneer u belt, vraag dan naar de Premier Support-verkoopmanager. Zie de [Veelgestelde vragen over het Microsoft Support Lifecycle-beleid](http://go.microsoft.com/fwlink/?linkid=169557) voor meer informatie.

Beveiligingslek met betrekking tot MX-records in SMTP-servers - CVE-2010-0024
-----------------------------------------------------------------------------

<span></span>
Er bestaat een beveiligingslek met betrekking tot denial of service door de wijze waarop het Microsoft Windows-onderdeel Simple Mail Transfer Protocol (SMTP) speciaal vervaardigde DNS MX-bronrecords verwerkt. Er is geen verificatie nodig om het beveiligingslek te kunnen misbruiken, waardoor een aanvaller dit beveiligingslek kan misbruiken door een speciaal vervaardigd netwerkbericht te verzenden naar een computer waarop de SMTP-service wordt uitgevoerd. Een aanvaller die dit lek weet te misbruiken, kan ervoor zorgen dat de SMTP-service niet meer reageert tot deze opnieuw wordt opgestart.

Zie [CVE 2010-0024](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-0024) als u dit beveiligingslek wilt weergeven als standaardvermelding in de lijst met veelvoorkomende beveiligingslekken en blootstellingen.

#### Beperkende factoren voor het beveiligingslek met betrekking tot MX-records in SMTP-servers - CVE-2010-0024

Met beperkende factoren wordt verwezen naar een instelling, de algehele configuratie of algemene tips voor een standaardstatus waarmee de ernst van misbruik via een beveiligingslek kan worden verminderd. De volgende beperkende factoren kunnen in uw situatie van pas komen:

-   Het SMTP-onderdeel wordt standaard niet geïnstalleerd op Windows Server 2003, Windows Server 2003 x64 Edition of Windows XP Professional x64 Edition.

#### Tijdelijke oplossingen voor het beveiligingslek met betrekking tot MX-records in SMTP-servers - CVE-2010-0024

Microsoft heeft nog geen oplossingen voor dit beveiligingslek.

#### Veelgestelde vragen over het beveiligingslek met betrekking tot MX-records in SMTP-servers - CVE-2010-0024

**Wat is de omvang van het beveiligingslek?**  
Dit beveiligingslek heeft betrekking op een denial of service. Een aanvaller die misbruik maakt van dit beveiligingslek, kan ervoor zorgen dat de SMTP-service niet meer reageert tot het speciaal vervaardigde bericht wordt verwijderd uit de wachtrij en de service handmatig opnieuw is opgestart. Het beveiligingslek betreft een 'denial-of-service'-probleem. Een aanvaller kan geen code uitvoeren of zichzelf gebruikersrechten toekennen, maar kan er wel voor zorgen dat het systeem waarin dit probleem optreedt geen aanvragen meer accepteert.

**Waardoor wordt het beveiligingslek veroorzaakt?**  
Het beveiligingslek bestaat doordat het Microsoft Windows-onderdeel Simple Mail Transfer Protocol (SMTP) een speciaal vervaardigd DNS MX-record (Mail Exchanger) onjuiste parseert.

**Wat is de SMTP-service?**  
De SMTP-service (Simple Mail Transfer Protocol) zorgt voor de overdracht van e-mail en wordt standaard geïnstalleerd als onderdeel van e-mailservices of Internet Information Services (IIS). SMTP bepaalt hoe e-mail over internet naar de doelserver wordt vervoerd en geleverd. De SMTP-service verzendt en ontvangt e-mail tussen servers. Zie het TechNet-artikel over de [SMTP-service (Simple Mail Transfer Protocol)](http://technet.microsoft.com/en-us/library/cc759281(ws.10).aspx) voor meer informatie.

**Wat is een MX-bronrecord?**  
De MX-bronrecord (Mail Exchange) geeft een e-mailserver op voor een DNS-domeinnaam. Een e-mailserver is een host die e-mail voor de DNS-domeinnaam verwerkt en doorstuurt. Verwerking van de e-mail betekent dat de e-mail wordt geleverd aan de geadresseerde of dat de e-mail wordt doorgestuurd naar een ander soort e-mailtransport. Het doorsturen van de e-mail betekent dat de e-mail wordt verzonden naar de doelserver, dat de e-mail wordt verzonden met behulp van SMTP (Simple Mail Transfer Protocol) naar een andere e-mailserver dichter bij de eindbestemming, of dat de e-mail een bepaalde tijd in een wachtrij wordt geplaatst. Zie het TechNet-artikel over [typen bronrecords](http://technet.microsoft.com/en-us/library/cc958958.aspx) voor meer informatie over MX-bronrecords.

**Met welk doel kan een aanvaller het beveiligingslek misbruiken?**  
Een aanvaller die misbruik weet te maken van dit beveiligingslek, kan ervoor zorgen dat de SMTP-service niet meer reageert tot het speciaal vervaardigde bericht wordt verwijderd uit de wachtrij en de service handmatig opnieuw is opgestart.

**Hoe kan een aanvaller dit beveiligingslek misbruiken?**  
Een aanvaller kan proberen misbruik te maken van het beveiligingslek door een schadelijke DNS-server te maken die een speciaal vervaardigde reactie op een query van een MX-bronrecord terugstuurt.

**Voor welke systemen vormt dit beveiligingslek het grootste risico?**  
Dit beveiligingslek vormt een risico voor alle besturingssystemen waarop het oorspronkelijke SMTP-onderdeel, dat als onderdeel van het besturingssysteem wordt geleverd, wordt gebruikt.

**Wat doet de update?**  
De beveiligingsupdate lost het beveiligingslek op door de manier te verbeteren waarop de SMTP-service MX-records parseert.

**Was dit beveiligingslek openbaar gemaakt op het moment dat dit beveiligingsbulletin werd uitgegeven?**  
Ja. Dit beveiligingslek is openbaar gemaakt. Aan dit beveiligingslek is nummer [CVE-2010-0024](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-0024) toegewezen.

**Was aan Microsoft gerapporteerd dat er misbruik was gemaakt van dit beveiligingslek op het moment dat dit beveiligingsbulletin werd uitgegeven?**  
Nee. Microsoft heeft geen informatie ontvangen die erop wees dat dit beveiligingslek was gebruikt om aanvallen uit te voeren op klanten op het moment dat dit beveiligingsbulletin oorspronkelijk werd uitgegeven.

Beveiligingslek met betrekking tot geheugentoewijzing in SMTP - CVE-2010- 0025
------------------------------------------------------------------------------

<span></span>
Er bestaat een beveiligingslek met betrekking tot het vrijgeven van informatie in het Microsoft Windows-onderdeel Simple Mail Transfer Protocol (SMTP) door de manier waarop het SMTP-onderdeel geheugen toewijst. Een aanvaller kan misbruik maken van het beveiligingslek door ongeldige opdrachten, gevolgd door de opdracht STARTTLS, te verzenden naar een server waarop dit probleem optreedt. Een aanvaller die misbruik weet te maken van dit beveiligingslek, kan fragmenten lezen van willekeurige e-mailberichten die zijn opgeslagen op de server waarop dit probleem optreedt. Dit beveiligingslek kan er niet toe leiden dat een aanvaller rechtstreeks code uitvoert of zich machtigingen toekent, maar wel dat de aanvaller bruikbare informatie kan achterhalen waarmee het systeem waarin dit probleem optreedt verder kan worden misbruikt.

Zie [CVE 2010-0025](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-0025) als u dit beveiligingslek wilt weergeven als standaardvermelding in de lijst met veelvoorkomende beveiligingslekken en blootstellingen.

#### Veelgestelde vragen over het beveiligingslek met betrekking tot geheugentoewijzing in SMTP - CVE-2010- 0025

**Wat is de omvang van het beveiligingslek?**  
Dit is een beveiligingslek met betrekking tot het vrijgeven van informatie. **Waardoor wordt het beveiligingslek veroorzaakt?**  
Het beveiligingslek bestaat doordat het Microsoft Windows-onderdeel Simple Mail Transfer Protocol (SMTP) onjuist geheugen toewijst bij het interpreteren van reacties op SMTP-opdrachten.

**Wat is de SMTP-service?**  
De SMTP-service (Simple Mail Transfer Protocol) zorgt voor de overdracht van e-mail en wordt standaard geïnstalleerd als onderdeel van e-mailservices of Internet Information Services (IIS). Zie het TechNet-artikel over de [SMTP-service (Simple Mail Transfer Protocol)](http://technet.microsoft.com/en-us/library/cc759281(ws.10).aspx) voor meer informatie.

**Wat is het TLS-protocol?**  
Het Transport Layer Security (TLS) handshake-protocol is verantwoordelijk voor de verificatie en de uitwisseling van sleutels die nodig is om veilige sessies te beginnen of te hervatten. Tijdens het opzetten van een veilige sessie beheert het handshake-protocol het volgende:

-   Onderhandeling over de coderingssuite
-   Verificatie van de server en optioneel van de client
-   Uitwisseling van belangrijke sessie-informatie

Zie het TechNet-artikel [How TLS/SSL Works](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) (Engelstalig) voor meer informatie.

**Wat is SSL?**  
Secure Sockets Layer (SSL) is een voorganger van het TLS-protocol. SSL voert dezelfde functies uit en ondersteunt beveiligde netwerkcommunicatie met een combinatie van technologie met openbare en geheime sleutels. Zie het TechNet-artikel [How TLS/SSL Works](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) (Engelstalig) voor meer informatie.

**Wat is het SChannel-beveiligingspakket?**  
Het beveiligingspakket Secure Channel (SChannel) is een Security Support Provider (SSP) waarmee de standaardverificatieprotocollen Secure Sockets Layer (SSL) en Transport Layer Security (TLS) worden uitgevoerd. Deze onderdelen worden gebruikt om veilige communicatie te implementeren ter ondersteuning van verschillende gangbare internet- en netwerktoepassingen, zoals websurfen. SChannel maakt deel uit van het beveiligingspakket dat bijdraagt aan een verificatieservice voor veilige communicatie tussen clients en servers. Zie het MSDN-artikel over het [Secure Channel](http://msdn.microsoft.com/en-us/library/aa380123(vs.85).aspx) voor meer informatie.

**Met welk doel kan een aanvaller het beveiligingslek misbruiken?**  
Een aanvaller die misbruik weet te maken van dit beveiligingslek, kan toegang krijgen tot fragmenten van willekeurige e-mailberichten op de server waarop dit probleem optreedt.

**Hoe kan een aanvaller dit beveiligingslek misbruiken?**  
Een aanvaller kan proberen misbruik te maken van het beveiligingslek door een stel speciaal vervaardigde SMTP-opdrachten naar een getroffen SMTP-server te verzenden. De aanvaller kan vervolgens informatie lezen in het geheugen van het getroffen systeem.

**Voor welke systemen vormt dit beveiligingslek het grootste risico?**

**Controleren of de update is toegepast**

-   **Microsoft Baseline Security Analyzer**

    Gebruik het hulpprogramma Microsoft Baseline Security Analyzer (MBSA) om te controleren of de beveiligingsupdate is toegepast op het systeem waarvoor het probleem geldt. Zie de sectie **Hulpmiddelen en richtlijnen voor detecteren en implementeren** eerder in dit bulletin voor meer informatie.

-   **Verificatie van bestandsversie**

    Omdat er verschillende versies zijn van Microsoft Windows, kunnen de volgende stappen voor uw computer enigszins afwijken. Als dit het geval is, raadpleeg dan de productdocumentatie om deze stappen uit te voeren.

    1.  Klik op **Start** en klik vervolgens op **Zoeken**.
    2.  Klik in het venster **Zoekresultaten** op **Alle bestanden en mappen** onder **Zoekassistent**.
    3.  Typ in het vak **De volledige of gedeeltelijke bestandsnaam** een bestandsnaam uit de betreffende bestandsinformatietabel en klik vervolgens op **Zoeken**.
    4.  Klik in de lijst met bestanden met de rechtermuisknop op een bestandsnaam uit de betreffende bestandsinformatietabel en klik vervolgens op **Eigenschappen**.
        **Opmerking** Bepaalde bestanden die in de bestandsinformatietabel staan vermeld, worden mogelijk niet geïnstalleerd. Dit is afhankelijk van de editie van het besturingssysteem en de programma's die op uw systeem zijn geïnstalleerd.
    5.  Bepaal op het tabblad **Versie** de versie van het bestand dat op uw systeem is geïnstalleerd door de versie te vergelijken met de versie in de betreffende bestandsinformatietabel.
        **Opmerking** Andere kenmerken dan de bestandsversie kunnen tijdens de installatie worden veranderd. Vergelijking van andere bestandskenmerken met de gegevens in de bestandsinformatietabel vormt geen ondersteunde methode om te controleren of de update is toegepast. Bovendien kunnen in bepaalde gevallen de namen van bestanden tijdens installatie worden gewijzigd. Als de bestands- of versiegegevens niet aanwezig zijn, gebruikt u een van de andere beschikbare methoden om te controleren of de update is geïnstalleerd.

-   **Verificatie van registersleutel**

    Aan de hand van de registersleutels in de **Referentietabel** in deze sectie kunt u ook controleren welke bestanden met deze beveiligingsupdate zijn geïnstalleerd.

    Deze registersleutels bevatten mogelijk geen volledige lijst met geïnstalleerde bestanden. Bovendien zijn deze registersleutels mogelijk niet correct als een beheerder of een OEM de beveiligingsupdate integreert in de bronbestanden van de Windows-installatie.
